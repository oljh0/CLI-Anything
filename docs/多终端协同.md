# CLI-Anything 跨终端协同任务系统

本项目致力为本地多终端环境提供一套统一的协同任务系统。采用 **主终端（Master）+ 子终端（Worker）** 协同模型：主终端负责发起任务、拆解子任务、监控进度并验收成果；子终端主动领取子任务进行开发，提交时自动运行测试并报告结果。系统内置 Web 可视化看板和终端 TUI 界面，让开发者随时掌握任务全貌。

## 🌟 核心特性

- **主从协同模型**：主终端发起并拆解任务，子终端主动领取子任务开发
- **任务全生命周期**：创建 → 拆解 → 领取 → 开发 → 测试 → 提交 → 验收
- **自动化测试集成**：子终端提交任务时自动运行单元测试，结果实时上报
- **主终端验收机制**：主终端校验测试结果，审核通过后子任务才标记完成
- **可视化看板**：Web 仪表盘（主要）+ 终端 TUI（辅助），实时展示任务进度与测试状态
- **双层架构**：底层 SQLite WAL 保证并发安全，上层 MCP Server 让 AI Agent 直接调用
- **零部署依赖**：全部在本机运行，无需额外服务器
- **AI Agent 集成**：通过 MCP 协议，Copilot CLI / Claude 等 AI 助手可直接参与任务协同

### 组件职责

| 组件 | 职责 | 技术选型 |
|------|------|----------|
| **CLI 层** | 解析命令行参数，格式化输出，角色化命令（master/worker） | [typer](https://typer.tiangolo.com/) + [rich](https://rich.readthedocs.io/) |
| **MCP Server 层** | 将任务管理能力暴露为 MCP 工具，供 AI Agent 调用 | [fastmcp](https://github.com/jlowin/fastmcp) |
| **Web Server** | 提供 REST API 和可视化看板页面 | [FastAPI](https://fastapi.tiangolo.com/) + 前端 (HTML/JS) |
| **Test Runner** | 子任务提交时自动发现并运行测试，收集结果 | pytest + 自定义 runner |
| **TaskManager Core** | 业务逻辑核心：CRUD、拆解、领取、状态流转、测试管理、验收 | 纯 Python |
| **Storage Layer** | 数据持久化，并发控制 | SQLite 3 (WAL 模式) |
| **TUI 界面** | 终端内的任务概览与状态查看 | [textual](https://textual.textualize.io/) |

    │    cli-anything verify <subtask_id> fail  │
    │    --reason "缺少边界测试"                 │
    │                                         │

## 📊 数据模型

### Task（任务）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | TEXT (UUID) | ✅ | 任务唯一标识，自动生成 |
| `title` | TEXT | ✅ | 任务标题 |
| `description` | TEXT | ❌ | 任务详细描述 |
| `status` | TEXT | ✅ | 任务状态，见状态机定义 |
| `task_type` | TEXT | ✅ | 任务类型：`master`（主任务）/ `subtask`（子任务） |
| `priority` | INTEGER | ❌ | 优先级 1-5，默认 3（1=最高） |
| `tags` | TEXT (JSON) | ❌ | 标签列表，JSON 数组格式 `["bug","urgent"]` |
| `parent_id` | TEXT | ❌ | 父任务 ID（子任务必填） |
| `created_by` | TEXT | ✅ | 创建者终端 ID（通常为主终端） |
| `claimed_by` | TEXT | ❌ | 领取者终端 ID（子终端领取后填入） |
| `claimed_at` | TEXT (ISO8601) | ❌ | 领取时间 |
| `submitted_at` | TEXT (ISO8601) | ❌ | 提交时间 |
| `verified_by` | TEXT | ❌ | 验收者终端 ID |
| `verified_at` | TEXT (ISO8601) | ❌ | 验收时间 |
| `verify_comment` | TEXT | ❌ | 验收备注（通过/打回原因） |
| `test_status` | TEXT | ❌ | 测试状态：`not_run` / `running` / `passed` / `failed` |
| `test_report` | TEXT (JSON) | ❌ | 测试报告摘要（JSON，含通过数/失败数/详情） |
| `test_path` | TEXT | ❌ | 关联的测试文件路径（相对项目根目录） |
| `work_dir` | TEXT | ❌ | 子任务工作目录路径 |
| `created_at` | TEXT (ISO8601) | ✅ | 创建时间 |
| `updated_at` | TEXT (ISO8601) | ✅ | 最后更新时间 |

**DDL：**
```sql
CREATE TABLE IF NOT EXISTS tasks (
    id              TEXT PRIMARY KEY,
    title           TEXT NOT NULL,
    description     TEXT DEFAULT '',
    status          TEXT NOT NULL DEFAULT 'pending',
    task_type       TEXT NOT NULL DEFAULT 'master',
    priority        INTEGER DEFAULT 3 CHECK (priority BETWEEN 1 AND 5),
    tags            TEXT DEFAULT '[]',
    parent_id       TEXT REFERENCES tasks(id) ON DELETE CASCADE,
    created_by      TEXT NOT NULL DEFAULT '',
    claimed_by      TEXT DEFAULT NULL,
    claimed_at      TEXT DEFAULT NULL,
    submitted_at    TEXT DEFAULT NULL,
    verified_by     TEXT DEFAULT NULL,
    verified_at     TEXT DEFAULT NULL,
    verify_comment  TEXT DEFAULT '',
    test_status     TEXT DEFAULT 'not_run',
    test_report     TEXT DEFAULT '{}',
    test_path       TEXT DEFAULT '',
    work_dir        TEXT DEFAULT '',
    created_at      TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at      TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_tasks_status ON tasks(status);
CREATE INDEX idx_tasks_type ON tasks(task_type);
CREATE INDEX idx_tasks_parent ON tasks(parent_id);
CREATE INDEX idx_tasks_claimed_by ON tasks(claimed_by);
CREATE INDEX idx_tasks_priority ON tasks(priority);
```

### TaskLog（任务操作日志）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | INTEGER | ✅ | 自增主键 |
| `task_id` | TEXT | ✅ | 关联任务 ID |
| `action` | TEXT | ✅ | 操作类型（见下表） |
| `terminal_id` | TEXT | ❌ | 执行操作的终端标识 |
| `old_value` | TEXT | ❌ | 变更前的值（JSON） |
| `new_value` | TEXT | ❌ | 变更后的值（JSON） |
| `detail` | TEXT | ❌ | 操作详情/备注 |
| `timestamp` | TEXT (ISO8601) | ✅ | 操作时间 |

**操作类型枚举：**

| action 值 | 说明 | 触发方 |
|-----------|------|--------|
| `created` | 任务创建 | 主终端 |
| `decomposed` | 主任务被拆解为子任务 | 主终端 |
| `claimed` | 子任务被领取 | 子终端 |
| `unclaimed` | 子任务被释放 | 子终端/主终端 |
| `status_changed` | 状态变更 | 任意终端 |
| `submitted` | 子任务提交（含测试结果） | 子终端 |
| `test_run` | 测试执行 | 系统自动 |
| `verified` | 验收通过 | 主终端 |
| `rejected` | 验收打回 | 主终端 |
| `updated` | 字段更新 | 任意终端 |
| `deleted` | 任务删除 | 主终端 |

**DDL：**
```sql
CREATE TABLE IF NOT EXISTS task_logs (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id     TEXT NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
    action      TEXT NOT NULL,
    terminal_id TEXT DEFAULT '',
    old_value   TEXT DEFAULT '',
    new_value   TEXT DEFAULT '',
    detail      TEXT DEFAULT '',
    timestamp   TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_task_logs_task ON task_logs(task_id);
CREATE INDEX idx_task_logs_action ON task_logs(action);
CREATE INDEX idx_task_logs_time ON task_logs(timestamp);
```

### 任务状态机

子任务相比主任务有更丰富的状态流转（含领取、提交、测试、验收）：

```
                    ┌─────────┐
                    │ pending │ ← 初始状态（可被领取）
                    └────┬────┘
                         │ claim（子终端领取）
                         ▼
                  ┌─────────────┐
           ┌──────│  claimed    │──────────┐
           │      └──────┬──────┘          │
           │ unclaim     │ start           │ cancel
           ▼             ▼                 ▼
     ┌─────────┐  ┌─────────────┐   ┌───────────┐
     │ pending │  │ in_progress │   │ cancelled │
     └─────────┘  └──────┬──────┘   └───────────┘
                         │
                         │ submit（提交 + 自动测试）
                         ▼
                  ┌─────────────┐
                  │  submitted  │ ← 等待验收
                  └──────┬──────┘
                    ┌────┴────┐
                    │         │
              verify:pass  verify:fail
                    │         │
                    ▼         ▼
             ┌──────────┐ ┌──────────┐
             │   done   │ │ rejected │
             └──────────┘ └────┬─────┘
                               │ 重新开发
                               ▼
                        ┌─────────────┐
                        │ in_progress │
                        └─────────────┘
```

**合法状态流转表：**

| 当前状态 | 可转换为 | 触发操作 |
|----------|----------|----------|
| `pending` | `claimed`, `cancelled` | 子终端领取 / 主终端取消 |
| `claimed` | `in_progress`, `pending`, `cancelled` | 开始开发 / 释放领取 / 取消 |
| `in_progress` | `submitted`, `blocked`, `cancelled` | 提交 / 阻塞 / 取消 |
| `submitted` | `done`, `rejected` | 主终端验收通过 / 打回 |
| `rejected` | `in_progress` | 子终端重新开发 |
| `blocked` | `in_progress`, `cancelled` | 解除阻塞 / 取消 |
| `done` | （终态） | — |
| `cancelled` | `pending`（可重新激活） | 主终端重新激活 |

## 🔌 接口契约

### CLI 命令清单

#### 通用命令
```bash
cli-anything init [--role master|worker] [--name "终端名称"]   # 初始化并注册终端
cli-anything --version                                          # 查看版本
cli-anything config show                                        # 查看配置
cli-anything config set <key> <value>                           # 修改配置
```

#### 主终端 (Master) 命令
```bash
# 任务创建与拆解
cli-anything create "任务标题" [--desc "描述"] [--priority 1-5] [--tags "tag1,tag2"]
cli-anything decompose <task_id> [--subtasks "子任务1;子任务2;子任务3"]
cli-anything decompose <task_id> --interactive        # 交互式拆解（逐个输入子任务）

# 任务监控
cli-anything list [--status ...] [--type master|subtask] [--claimed-by <terminal_id>]
cli-anything show <task_id>                           # 查看任务详情（含子任务列表）
cli-anything progress [<task_id>]                     # 查看主任务完成进度

# 验收审核
cli-anything review <subtask_id>                      # 查看子任务提交内容和测试报告
cli-anything verify <subtask_id> pass                 # 验收通过
cli-anything verify <subtask_id> fail --reason "打回原因"  # 验收打回

# 可视化看板
cli-anything dashboard [--port 8080]                  # 启动 Web 看板
cli-anything tui                                      # 启动终端 TUI 界面

# MCP Server
cli-anything serve [--transport stdio|sse] [--port 8000]

# 日志与管理
cli-anything log [<task_id>] [--limit 20] [--action claimed|submitted|verified]
cli-anything delete <task_id> [--force]               # 删除任务（级联删除子任务）
```

#### 子终端 (Worker) 命令
```bash
# 查看与领取
cli-anything available [--tag "标签"] [--sort priority]  # 查看可领取的子任务
cli-anything claim <subtask_id>                          # 领取子任务
cli-anything unclaim <subtask_id>                        # 释放已领取但未开始的子任务
cli-anything my                                          # 查看我领取的所有子任务

# 开发过程
cli-anything start <subtask_id>                          # 标记开始开发
cli-anything status <subtask_id> blocked [--reason "原因"]  # 标记阻塞

# 提交与测试
cli-anything submit <subtask_id> [--test-path "tests/test_xxx.py"]
#   → 自动运行 pytest 对指定测试文件
#   → 收集测试结果并更新 test_status / test_report
#   → 将状态变更为 submitted

cli-anything test <subtask_id> [--test-path "tests/test_xxx.py"]  # 仅运行测试，不提交

# 查看反馈
cli-anything feedback <subtask_id>                      # 查看验收反馈
```

  "parameters": {

## 🧪 测试验证机制

### 自动测试流程

```
子终端执行 submit 命令
       │
       ▼
检查是否指定了 test_path
       │
       ├─→ 有：使用指定路径
       └─→ 无：尝试从任务 test_path 字段读取
              │
              └─→ 无：跳过测试，直接提交（标记 test_status = not_run）
       │
       ▼
运行 pytest（带超时控制）
       │
       ▼
收集测试结果
  ├─ 通过数、失败数、跳过数、错误数
  ├─ 总耗时
  ├─ 失败用例详情
  └─ 输出日志
       │
       ▼
更新任务记录
  ├─ test_status = passed / failed
  ├─ test_report = { ... JSON 详情 ... }
  └─ status = submitted
       │
       ▼
写入 task_logs（action = submitted + test_run）
       │
       ▼
通知主终端（如已配置）
```

### 测试报告格式

```json
{
  "test_status": "passed",
  "summary": {
    "total": 12,
    "passed": 11,
    "failed": 1,
    "skipped": 0,
    "errors": 0,
    "duration": 3.45
  },
  "failures": [
    {
      "test_name": "test_user_register_duplicate_email",
      "file": "tests/test_user_register.py",
      "line": 42,
      "message": "AssertionError: Expected 409, got 200"
    }
  ],
  "timestamp": "2026-04-10T20:00:00"
}
```

### 主终端验收规则

- 测试全部通过 → 主终端可直接通过验收
- 测试有失败 → 看板标红提醒，主终端审查后决定通过或打回
- 无测试 → 看板标黄提醒，建议补充测试后再验收

### Skill 1: master-terminal.md（主终端技能）

```markdown
# 主终端 (Master Terminal) — 任务编排与质量把控

你是 CLI-Anything 系统中的**主终端 AI 助手**，负责任务的全生命周期编排与质量把控。

## 角色定位

你是一名**技术项目经理 + 质量守门人**，核心职责：
1. 理解用户需求，创建结构清晰的主任务
2. 将主任务合理拆解为可独立执行的子任务
3. 监控各子任务进度，识别阻塞和风险
4. 审核子终端提交的成果和测试报告
5. 做出验收决策（通过/打回并给出具体改进意见）

## 可用工具

你可以通过 MCP 协议调用以下 cli-anything 工具：

| 工具 | 用途 | 使用场景 |
|------|------|----------|
| `task_create` | 创建主任务 | 用户描述需求时 |
| `task_decompose` | 拆解子任务 | 主任务创建后，分析需求并拆解 |
| `task_list` | 查询任务列表 | 检查整体进度、筛选特定状态任务 |
| `task_progress` | 查看主任务进度 | 汇报进度、识别瓶颈 |

| `task_log` | 查询操作日志 | 追溯操作历史 |

## 工作流程

### 1. 需求分析与任务创建
当用户提出需求时：
- 先澄清需求范围和验收标准
- 创建主任务，标题简明，描述包含明确的验收标准
- 设置合理的优先级和标签

### 2. 任务拆解原则
拆解子任务时遵循以下原则：
- **独立性**：每个子任务应尽量独立，减少相互依赖
- **可测试性**：每个子任务必须有明确的测试验收条件
- **适当粒度**：每个子任务应在 1-4 小时内可完成
- **清晰描述**：子任务描述必须包含：
  - 具体实现要求
  - 预期输入/输出
  - 测试要求（需要测试哪些场景）
  - 相关文件路径提示

### 3. 进度监控
定期检查进度时关注：
- 哪些子任务长时间未被领取（考虑调整优先级或描述）
- 哪些子任务处于阻塞状态（主动协助解决）
- 整体完成率与预期是否匹配

### 4. 验收审核标准
审核子任务提交时：
- **测试通过率**：所有测试必须通过，测试覆盖关键场景
- **代码质量**：检查实现是否符合描述中的要求
- **边界处理**：错误输入、空值、并发等边界场景
- **打回时**：必须给出具体、可操作的改进建议，而非模糊反馈

## 行为约束

- ❌ 不要自己编写实现代码（那是子终端的工作）
- ❌ 不要跳过测试审核直接通过验收
- ❌ 不要一次性拆解过多子任务（建议 3-7 个为佳）
- ✅ 任务描述要足够详细，让子终端无需额外沟通即可开始工作
- ✅ 打回时提供具体的改进方向和示例
- ✅ 定期用 task_progress 检查整体进度
```

| `task_update` | 更新任务信息 | 补充工作目录、测试路径等 |

同时你拥有完整的开发工具链：文件读写、终端命令、代码搜索等。

## 工作流程

### 1. 选择任务
- 用 `task_list` 查看 `status=pending` 的子任务
- 阅读任务描述，评估自己是否具备完成能力
- 优先选择高优先级任务
- 用 `task_claim` 领取选定的任务

### 2. 理解需求
领取任务后，在开始编码前：
- 仔细阅读任务描述中的实现要求和验收标准
- 分析任务涉及的现有代码（如有）
- 确定实现方案和文件结构
- 如有不清楚的地方，在任务日志中记录疑问

### 3. 编写实现代码
- 遵循项目的代码规范（见 shared-conventions.md）
- 编写清晰、可维护的代码
- 添加必要的注释（中文注释优先）
- 处理边界情况和错误场景

### 4. 编写单元测试（必须）
每个子任务**必须**包含对应的单元测试：

**测试文件命名**：`tests/test_<模块名>.py`

**测试覆盖要求**：
- 正常流程测试（happy path）
- 边界条件测试（空输入、极大/极小值、特殊字符）
- 错误处理测试（无效输入、异常场景）
- 如涉及数据库操作，使用内存 SQLite 或临时文件

**测试代码规范**：
```python
import pytest

class TestXxx:
    """测试 Xxx 功能"""

    def test_normal_case(self):
        """正常流程：xxx"""
        ...

    def test_boundary_empty_input(self):
        """边界条件：空输入应返回 xxx"""
        ...

    def test_error_invalid_param(self):
        """错误处理：无效参数应抛出 ValueError"""
        ...
```

### 5. 验证与提交
- 在本地运行全部测试，确保通过：`pytest tests/test_xxx.py -v`
- 使用 `task_submit` 提交（系统会自动再次运行测试）
- 如测试失败，修复后重新提交

### 6. 处理验收反馈
如果任务被打回（rejected）：
- 用 `task_log` 查看打回原因
- 根据具体反馈进行修改
- 修改完成后重新提交

## 行为约束

- ❌ 不要在没有测试的情况下提交任务
- ❌ 不要修改其他子任务的代码范围
- ❌ 不要跳过本地测试直接提交
- ✅ 每个子任务提交前，本地测试必须全部通过
- ✅ 测试覆盖正常流程 + 至少 2 个边界/错误场景
- ✅ 代码提交前检查是否符合 shared-conventions.md 中的规范
- ✅ 如遇阻塞，及时更新任务状态为 blocked 并说明原因
```

### Skill 3: shared-conventions.md（共享约定）

```markdown
# 项目共享约定

本文件定义 CLI-Anything 项目的通用规范，主终端和子终端均需遵守。

## 代码规范

### Python 代码风格
- 遵循 PEP 8

- 文件名：snake_case（如 `task_manager.py`）
- 类名：PascalCase（如 `TaskManager`）

- 常量：UPPER_SNAKE_CASE（如 `DEFAULT_PRIORITY`）

### 注释与文档
- 关键逻辑添加中文注释
- 公开 API 必须有文档字符串
- 复杂算法需注释说明思路

## 任务规范

### 任务标题
- 使用动词开头：「实现 xxx」「修复 xxx」「优化 xxx」「添加 xxx」

### 技能与 MCP Server 的集成

Skills 通过以下方式与 CLI-Anything 系统集成：

```
┌───────────────────────────────────────────┐
│         Copilot CLI / Claude Agent         │
│                                           │
│  ┌─────────────────┐  ┌────────────────┐  │
│  │ Skill 文件加载   │  │ MCP Client     │  │
│  │ (角色行为引导)   │  │ (工具调用)     │  │
│  └────────┬────────┘  └───────┬────────┘  │
│           │                   │           │
│           ▼                   ▼           │
│  Agent 按角色行为         调用 MCP 工具     │
│  理解任务/写代码           读写任务数据     │
└───────────────────────────────────────────┘
                    │
                    ▼
         ┌──────────────────┐
         │ cli-anything     │
         │ MCP Server       │
         │ (fastmcp)        │
         └────────┬─────────┘
                  │
                  ▼
         ┌──────────────────┐
         │ SQLite (tasks.db)│
         └──────────────────┘
```

**自动加载机制**：
1. Agent 启动时读取 `~/.cli-anything/config.yaml` 中的 `terminal.role`
2. 根据角色加载对应的 Skill 文件到 Agent 上下文
3. Skill 中定义的行为约束引导 Agent 使用正确的工具和工作流程
4. MCP Server 提供实际的数据读写能力
