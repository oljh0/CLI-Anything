## 1. 模块概述

CLI-Anything 提供两个核心辅助模块，分别处理运行时健康监控与自动化测试：

| 模块 | 职责 |
|------|------|
| **TerminalHealthChecker** | 终端心跳检测、过期终端识别、过期领取自动释放 |
| **测试运行器**（`run_tests` / `run_tests_simple`） | 集成 pytest 执行测试，解析 JSON 报告或标准输出，返回结构化 `TestReport` |

两个模块均依赖核心层（`Database` / `TaskManager`），并通过 CLI 命令和 MCP 工具对外暴露。

```python
class TerminalHealthChecker:
```


| 参数 | 类型 | 默认值 | 说明 |

- `self.db` — 数据库引用
- `self.timeout_seconds` — 超时阈值

### 方法详解

#### `check_alive(terminal_id: str) → bool`

检查指定终端是否仍然在线。

**判断逻辑：**

1. 通过 `self.db.get_terminal(terminal_id)` 获取终端记录
2. 若终端不存在或 `last_active` 为空 → 返回 `False`
3. 将 `last_active`（ISO 格式字符串）解析为 `datetime` 对象
4. 计算 `当前时间 - last_active` 的总秒数
5. 若差值 **小于** `timeout_seconds` → 返回 `True`（在线）
6. 若差值 **大于等于** `timeout_seconds` → 返回 `False`（过期）
7. 若 `last_active` 格式解析失败 → 返回 `False`

```python
try:
    last = datetime.fromisoformat(terminal.last_active)
    return (datetime.now() - last).total_seconds() < self.timeout_seconds
except ValueError:
    return False
```

**返回值示例：**
```python
checker.check_alive("terminal-abc")  # True  — 终端活跃
checker.check_alive("terminal-xyz")  # False — 终端超时或不存在
```

---

#### `list_stale_terminals() → list[Terminal]`

获取所有已超时的终端列表。

**流程：**

1. 调用 `self.db.list_terminals()` 获取全部注册终端
2. 逐一调用 `self.check_alive(t.id)` 检测
3. 将 **未通过** 检测的终端收集到 `stale` 列表

```python
all_terminals = self.db.list_terminals()
stale = []
for t in all_terminals:
    if not self.check_alive(t.id):
        stale.append(t)
return stale
```

**返回值：** `Terminal` 对象列表，每个元素代表一个过期终端。

---

#### `cleanup_stale_claims() → list[dict]`

核心清理方法：查找被过期终端领取的任务，将其释放回 `PENDING` 状态。

**流程：**

1. 调用 `self.list_stale_terminals()` 获取过期终端集合
2. 若无过期终端，直接返回空列表
3. 创建临时 `TaskManager` 实例（`terminal_id="health-checker"`）用于记录日志
4. 遍历每个过期终端 ID：
   - 调用 `self.db.list_tasks(claimed_by=terminal_id)` 查找该终端领取的任务
   - 对状态为 `CLAIMED` 或 `IN_PROGRESS` 的任务执行释放：
     - 将 `status` 设为 `TaskStatus.PENDING`
     - 清空 `claimed_by` 和 `claimed_at`
     - 调用 `self.db.update_task(task)` 持久化
     - 通过 `tm._log()` 记录操作日志
5. 返回所有被释放任务的信息列表

**释放逻辑中的关键状态判断：**

```python
if task.status in (TaskStatus.CLAIMED, TaskStatus.IN_PROGRESS):
    task.status = TaskStatus.PENDING
    task.claimed_by = None
    task.claimed_at = None
```

> 只有 `CLAIMED`（已领取但未开始）和 `IN_PROGRESS`（进行中）两种状态的任务会被释放。其他状态（如 `DONE`、`CANCELLED`）不受影响。

**返回值结构：**

```python
[
    {
        "task_id": "task-001",
        "title": "实现用户登录",
        "old_status": "claimed",        # 释放前的状态
        "stale_terminal": "terminal-abc" # 过期终端 ID
    },
    # ...
]
```

**日志记录：**

每次释放都会通过 `TaskManager._log()` 写入操作记录：

```
终端 terminal-abc 超时，自动释放任务
```

---

#### `heartbeat(terminal_id: str) → None`

更新终端心跳时间，用于维持终端在线状态。

**流程：**

1. 通过 `self.db.get_terminal(terminal_id)` 获取终端
2. 若终端存在，将 `last_active` 更新为当前时间（`_now_iso()`）
3. 调用 `self.db.upsert_terminal(terminal)` 持久化

```python
def heartbeat(self, terminal_id: str):
    terminal = self.db.get_terminal(terminal_id)
    if terminal:
        terminal.last_active = _now_iso()
        self.db.upsert_terminal(terminal)
```

> 终端应定期调用此方法以避免被判定为过期。

---

### 设计要点

1. **基于时间戳的健康判断** — 通过 `last_active` 与当前时间的差值来确定终端是否在线，简单可靠
2. **自动清理机制** — `cleanup_stale_claims()` 防止任务因终端崩溃或断线而永久锁定在 `CLAIMED`/`IN_PROGRESS` 状态
3. **双重状态释放** — 同时处理 `CLAIMED` 和 `IN_PROGRESS` 状态，覆盖"领取后未开始"和"执行中断线"两种场景
4. **日志可追溯** — 每次自动释放都记录到任务日志，便于事后审计
5. **心跳续约** — `heartbeat()` 方法供终端定期调用，维持在线状态
6. **惰性导入** — `cleanup_stale_claims()` 内部惰性导入 `TaskStatus` 和 `TaskManager`，避免循环依赖
7. **可通过 CLI 触发** — 通过 `task health --cleanup` 命令或 MCP `task_health` 工具暴露

### TestReport 数据类

```python
@dataclass
class TestReport:
    """测试运行报告"""
```

**字段定义：**

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `total` | `int` | `0` | 测试用例总数 |
| `passed` | `int` | `0` | 通过数 |
| `failed` | `int` | `0` | 失败数 |
| `errors` | `int` | `0` | 错误数（如 fixture 异常） |
| `skipped` | `int` | `0` | 跳过数 |
| `duration` | `float` | `0.0` | 总耗时（秒） |
| `exit_code` | `int` | `0` | pytest 退出码 |
| `output` | `str` | `""` | pytest 标准输出 + 标准错误（合并） |
| `details` | `list[dict]` | `[]` | 失败/错误用例详情列表 |

**方法：**

#### `to_dict() → dict`

将报告序列化为字典（不含 `output` 字段），适合 JSON 序列化和 API 返回。

```python
{
    "total": 10,
    "passed": 8,
    "failed": 1,
    "errors": 1,
    "skipped": 0,
    "duration": 3.45,
    "exit_code": 1,
    "details": [...]
}
```

> **注意**：`duration` 在序列化时会被 `round(..., 2)` 保留两位小数。

#### `success` 属性（`@property`）

```python
@property
def success(self) -> bool:
    return self.exit_code == 0 and self.failed == 0 and self.errors == 0
```

三个条件同时满足才为 `True`：
- pytest 退出码为 0
- 无失败用例
- 无错误用例

### 核心函数：`run_tests()`

```python
def run_tests(
    test_path: str = "tests/",
    work_dir: Optional[str] = None,
    extra_args: Optional[list[str]] = None,
    timeout: int = 300,
) -> TestReport
```

**参数说明：**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `test_path` | `str` | `"tests/"` | 测试文件或目录路径 |
| `work_dir` | `Optional[str]` | `None` | 工作目录，`None` 时使用当前目录 `"."` |
| `extra_args` | `Optional[list[str]]` | `None` | 额外传递给 pytest 的参数列表 |
| `timeout` | `int` | `300` | 超时秒数（默认 5 分钟） |

**执行流程：**

```
1. 创建空 TestReport
2. 创建临时 JSON 文件路径
3. 构造 pytest 命令
4. subprocess.run() 执行
5. 解析 JSON 报告 → 填充 TestReport
6. 清理临时文件
7. 返回 TestReport
```

**步骤详解：**

**① 构造 pytest 命令**

```python
cmd = [
    sys.executable, "-m", "pytest",  # 使用当前 Python 解释器
    test_path,                        # 测试路径
    "-v",                             # 详细输出
    "--tb=short",                     # 简短 traceback
    f"--json-report-file={json_path}",# JSON 报告输出路径
    "--json-report",                  # 启用 JSON 报告插件
]
if extra_args:
    cmd.extend(extra_args)            # 附加额外参数
```

> 使用 `sys.executable -m pytest` 而非直接 `pytest` 命令，确保使用当前环境的 Python 解释器。

**② 子进程执行**

```python
result = subprocess.run(
    cmd,
    capture_output=True,  # 捕获 stdout + stderr
    text=True,            # 文本模式
    cwd=cwd,              # 工作目录
    timeout=timeout,      # 超时
)
```

**③ JSON 报告解析（主路径）**

当 `pytest-json-report` 插件可用时，从 JSON 文件中提取：

```python
data = json.loads(json_file.read_text(encoding="utf-8"))
summary = data.get("summary", {})
report.total    = summary.get("total", 0)
report.passed   = summary.get("passed", 0)
report.failed   = summary.get("failed", 0)
report.errors   = summary.get("error", 0)     # 注意: JSON 中键名为 "error"（单数）
report.skipped  = summary.get("skipped", 0)
report.duration = data.get("duration", report.duration)
```

**失败用例详情提取：**

```python
for test in data.get("tests", []):
    if test.get("outcome") in ("failed", "error"):
        report.details.append({
            "name": test.get("nodeid", ""),       # 测试节点 ID
            "outcome": test.get("outcome", ""),    # "failed" 或 "error"
            "message": test.get("call", {}).get("longrepr", ""),  # 详细错误信息
        })
```

> 只收集失败和错误用例的详情，通过的用例不记录详情。

**④ 降级路径（JSON 插件不可用）**

若 JSON 报告文件不存在，调用 `_parse_output_fallback()` 从 pytest 标准输出中提取结果。

**⑤ 超时处理**

```python
except subprocess.TimeoutExpired:
    report.duration = time.time() - start
    report.exit_code = -1
    report.output = f"测试超时（{timeout}秒）"
```

超时时 `exit_code` 设为 `-1`，`output` 记录超时信息。

**⑥ 临时文件清理**

```python
finally:
    try:
        Path(json_path).unlink(missing_ok=True)
    except OSError:
        pass
```

无论成功与否，始终尝试清理临时 JSON 文件。

---

### 辅助函数：`_parse_output_fallback()`

```python
def _parse_output_fallback(report: TestReport, result: subprocess.CompletedProcess)
```

当 `pytest-json-report` 插件未安装时的降级方案。从 pytest 标准输出中解析结果摘要。

**解析逻辑：**

1. 从 `result.stdout` 的最后几行**倒序搜索**摘要行
2. 匹配模式：类似 `5 passed, 2 failed in 1.23s` 的摘要格式
3. 使用正则表达式 `(\d+)\s+(passed|failed|error|skipped|warnings?)` 提取各项计数
4. 使用正则表达式 `in\s+([\d.]+)s` 提取耗时
5. 计算 `total = passed + failed + errors + skipped`

```python
for line in reversed(output.splitlines()):
    line = line.strip()
    if "passed" in line or "failed" in line or "error" in line:
        # 正则提取各指标 ...
        break  # 只解析第一个匹配行
```

> 这是一种 best-effort 解析：可能无法覆盖所有 pytest 输出格式，但能处理绝大多数标准场景。

### 简化入口：`run_tests_simple()`

```python
def run_tests_simple(
    test_path: str = "tests/",
    work_dir: Optional[str] = None,
    timeout: int = 300,
) -> TestReport
```

**与 `run_tests()` 的区别：**

| 特性 | `run_tests()` | `run_tests_simple()` |
|------|---------------|----------------------|
| JSON 报告插件 | **需要** `pytest-json-report` | **不需要** |
| 额外参数 | 支持 `extra_args` | 不支持 |
| pytest 参数 | `-v --tb=short --json-report --json-report-file=...` | `-v --tb=short` |
| 结果解析 | JSON 报告解析 → 降级到标准输出 | **仅**标准输出解析 |
| 失败详情 | 有（从 JSON 提取） | 无 |

**适用场景：**
- 环境中未安装 `pytest-json-report` 插件
- 只需要通过/失败的基本统计
- 无需失败用例的详细错误信息

**命令构造：**

```python
cmd = [sys.executable, "-m", "pytest", test_path, "-v", "--tb=short"]
```

执行和超时处理逻辑与 `run_tests()` 一致，结果解析固定使用 `_parse_output_fallback()`。

### pytest 集成细节

#### 依赖要求

| 依赖 | 必须 | 用途 |
|------|------|------|
| `pytest` | ✅ 是 | 测试框架 |
| `pytest-json-report` | ❌ 可选 | JSON 结构化报告（`run_tests()` 使用） |

#### pytest 退出码对照

| 退出码 | 含义 |
|--------|------|
| `0` | 全部通过 |
| `1` | 有测试失败 |
| `2` | 测试被中断 |
| `3` | 内部错误 |
| `4` | 命令行参数错误 |
| `5` | 未找到测试用例 |
| `-1` | 超时（CLI-Anything 自定义） |

#### 进程隔离

- 使用 `subprocess.run()` 在子进程中执行 pytest，避免测试代码污染主进程状态
- `capture_output=True` 捕获所有输出，不会干扰主进程的标准输出
- `text=True` 确保输出为字符串而非字节流

#### 工作目录

- 通过 `cwd` 参数设置 pytest 的工作目录
- 默认为 `"."`（当前目录）
- 可设置为任务的 `work_dir` 以在正确的项目目录中运行测试

---

### 设计要点

1. **双模式运行** — `run_tests()` 提供完整 JSON 报告支持，`run_tests_simple()` 作为无依赖降级方案
2. **优雅降级** — 即使 `pytest-json-report` 不可用，`run_tests()` 也能自动 fallback 到标准输出解析
3. **结构化结果** — `TestReport` 数据类提供统一的结果表示，支持 `to_dict()` 序列化和 `success` 属性判断
4. **进程隔离** — 通过 `subprocess` 运行测试，避免测试代码影响主进程
5. **超时保护** — 默认 300 秒超时，防止测试无限挂起
6. **失败详情收集** — JSON 模式下提取每个失败用例的 `nodeid`、`outcome` 和 `longrepr`，方便定位问题
7. **临时文件管理** — 使用 `tempfile.NamedTemporaryFile` 创建临时 JSON 文件，`finally` 块中确保清理
8. **错误处理覆盖** — 超时（`TimeoutExpired`）、JSON 解析失败（`JSONDecodeError`）、文件不存在等场景均有处理

### 暴露方式

| 模块 | CLI 命令 | MCP 工具 |
|------|---------|----------|
| TerminalHealthChecker | `task health [--cleanup]` | `task_health` |
| 测试运行器 | `task test <task_id> [--path]` | `task_test` |

### 依赖关系

- **TerminalHealthChecker** → `Database`（查询终端和任务）、`TaskManager`（记录日志）、`TaskStatus`（状态枚举）
- **测试运行器** → `subprocess`（执行 pytest）、`pytest` + `pytest-json-report`（可选）
- 两个模块之间 **无直接依赖**，各自独立工作

### 典型使用场景

#### 场景 1：定期健康检查

```
用户执行 `task health --cleanup`
  → TerminalHealthChecker.list_stale_terminals()
    → 找到 2 个过期终端
  → TerminalHealthChecker.cleanup_stale_claims()
    → 释放 3 个被锁定的任务 → 状态改为 PENDING
    → 其他活跃终端可重新领取这些任务
```

#### 场景 2：运行任务测试

```
用户执行 `task test task-001 --path tests/test_login.py`
  → run_tests("tests/test_login.py", work_dir="./project")
    → subprocess 执行 pytest
    → 解析 JSON 报告
    → 返回 TestReport(total=5, passed=4, failed=1, ...)
  → 更新任务的测试状态和报告
```

#### 场景 3：无 JSON 插件环境

```
run_tests_simple("tests/")
  → subprocess 执行 pytest（不带 --json-report）
  → 从标准输出解析 "5 passed, 1 failed in 2.34s"
  → 返回 TestReport(total=6, passed=5, failed=1, duration=2.34)
```
