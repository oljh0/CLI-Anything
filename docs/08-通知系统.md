## 1. 模块概述

`Notifier` 是 CLI-Anything 的 **跨平台桌面通知系统**，支持 Windows / macOS / Linux 三平台原生通知推送。

设计目标是在任务状态发生变更时，通过操作系统原生通知机制向用户发送提醒。通知模块不依赖任何第三方 Python 库，完全通过系统命令（PowerShell / osascript / notify-send）实现。

### 设计定位

```
┌────────────────────────────────────────┐
│  业务层: TaskManager                    │
├────────────────────────────────────────┤
│  Notifier  ← 本模块（通知推送）         │
├────────────────────────────────────────┤
│  OS 原生通知: PowerShell / osascript /  │
│               notify-send              │
└────────────────────────────────────────┘
```

### 集成状态

> ✅ **已完成核心集成** — Notifier 已通过构造注入方式集成到 TaskManager 主流程中。
>
> CLI 的 `_init()` 方法会自动创建 `Notifier` 实例并注入 `TaskManager`，状态变更、任务提交、验收操作均会自动触发桌面通知回调。所有通知调用均通过 `if self.notifier:` 守卫，确保未注入时不影响原有逻辑。
>
> 默认配置中 `notification.enabled` 为 `False`，需用户手动启用后通知才会实际发送。

## 3. 核心方法

### 3.1 notify(title, message, urgency)

```python
def notify(self, title: str, message: str, urgency: str = "normal")
```

**发送桌面通知的统一入口。**

| 参数      | 类型  | 默认值     | 说明                                |
| --------- | ----- | ---------- | ----------------------------------- |
| `title`   | `str` | —          | 通知标题                             |
| `message` | `str` | —          | 通知正文内容                          |
| `urgency` | `str` | `"normal"` | 紧急程度：`low` / `normal` / `critical` |

**执行逻辑**：

```
notify() 被调用
  │
  ├─ self._enabled == False → 直接返回，不发送
  │
  ├─ 读取 notification.type 配置（默认 "toast"，当前未使用）
  │
  └─ 根据 sys.platform 分发：
       ├─ "win32"  → _notify_windows(title, message)
       ├─ "darwin" → _notify_macos(title, message)
       └─ 其他     → _notify_linux(title, message, urgency)
```

**注意事项**：

- `urgency` 参数仅在 Linux 平台生效（`notify-send --urgency=`），Windows 和 macOS 平台忽略该参数
- `notification.type` 配置已读取但当前版本未使用，预留给未来扩展（如区分 toast / email / webhook 等通知类型）

## 4. 事件回调方法

Notifier 提供三个语义化的事件回调方法，对应任务生命周期的三个关键节点。每个方法内部都先检查对应的配置开关，再调用 `notify()` 发送通知。

### 4.1 on_status_change(task_id, old_status, new_status, title)

```python
def on_status_change(self, task_id: str, old_status: str, new_status: str, title: str)
```

**任务状态变更通知。**

| 参数         | 类型  | 说明          |
| ------------ | ----- | ------------- |
| `task_id`    | `str` | 任务 ID       |
| `old_status` | `str` | 变更前的状态   |
| `new_status` | `str` | 变更后的状态   |
| `title`      | `str` | 任务标题       |

- **配置开关**：`notification.on_status_change`（默认 `True`）
- **通知标题**：`任务状态变更`
- **通知正文**：`[{task_id}] {title}\n{old_status} → {new_status}`

### 4.2 on_submit(task_id, title, terminal_id)

```python
def on_submit(self, task_id: str, title: str, terminal_id: str)
```

**任务提交通知。**

| 参数          | 类型  | 说明              |
| ------------- | ----- | ----------------- |
| `task_id`     | `str` | 任务 ID           |
| `title`       | `str` | 任务标题           |
| `terminal_id` | `str` | 提交任务的终端 ID  |

- **配置开关**：`notification.on_submit`（默认 `True`）
- **通知标题**：`📤 任务已提交`
- **通知正文**：`[{task_id}] {title}\n来自终端: {terminal_id}`

### 4.3 on_verify(task_id, title, approved, comment)

```python
def on_verify(self, task_id: str, title: str, approved: bool, comment: str)
```

**验收结果通知。**

| 参数       | 类型   | 说明                          |
| ---------- | ------ | ----------------------------- |
| `task_id`  | `str`  | 任务 ID                       |
| `title`    | `str`  | 任务标题                       |
| `approved` | `bool` | 是否通过验收                    |
| `comment`  | `str`  | 验收评语（可为空字符串）         |

- **配置开关**：`notification.on_verify`（默认 `True`）
- **通知标题**：`验收结果: ✅ 通过` 或 `验收结果: ❌ 驳回`
- **通知正文**：`[{task_id}] {title}\n{comment}`（comment 为空时省略换行和评语）
- **紧急程度**：驳回时 `urgency="critical"`，通过时 `urgency="normal"`

## 5. 平台实现

三个平台实现方法均为 `@staticmethod`，不依赖实例状态。所有平台实现共享以下设计原则：

- 使用 `subprocess.run()` 调用系统命令
- 统一设置 `capture_output=True` 抑制子进程输出
- 统一设置 `timeout=5` 秒防止挂起
- 所有异常静默捕获（`except Exception: pass`），**通知失败不影响主流程**

### 5.1 _notify_windows(title, message)

```python
@staticmethod
def _notify_windows(title: str, message: str)
```

**Windows Toast 通知实现**，采用两级 fallback 策略：

#### 主方案：WinRT Toast API

通过 PowerShell 调用 Windows Runtime 的 `ToastNotificationManager`：

```powershell
# 加载 WinRT 类型
[Windows.UI.Notifications.ToastNotificationManager, ...]  | Out-Null
[Windows.Data.Xml.Dom.XmlDocument, ...]                   | Out-Null

# 构造 Toast XML 模板
$template = @"
<toast>
    <visual><binding template="ToastGeneric">
        <text>{title}</text>
        <text>{message}</text>
    </binding></visual>
</toast>
"@

# 解析 XML 并发送通知
$xml = New-Object Windows.Data.Xml.Dom.XmlDocument
$xml.LoadXml($template)
$toast = [Windows.UI.Notifications.ToastNotification]::new($xml)
[Windows.UI.Notifications.ToastNotificationManager]::CreateToastNotifier("CLI-Anything").Show($toast)
```

- 通知来源标识为 `"CLI-Anything"`
- 使用 `ToastGeneric` 模板，包含标题和正文两行文本
- 要求 Windows 10+ 且 PowerShell 5.1+

#### Fallback 方案：MessageBox

当 WinRT API 不可用时（如旧版 Windows），回退到 `System.Windows.Forms.MessageBox`：

```powershell
[System.Windows.Forms.MessageBox]::Show("{message}", "{title}")
```

> ⚠️ **注意**：MessageBox 是阻塞式弹窗，不是 Toast 通知。在无人值守场景下可能导致进程挂起（虽然有 5 秒 timeout 保护）。

### 5.2 _notify_macos(title, message)

```python
@staticmethod
def _notify_macos(title: str, message: str)
```

**macOS 通知实现**，使用 `osascript` 调用 AppleScript：

```bash
osascript -e 'display notification "{message}" with title "{title}"'
```

- 依赖系统自带的 `osascript` 命令，无需安装额外软件
- 通知显示在 macOS 通知中心

### 5.3 _notify_linux(title, message, urgency)

```python
@staticmethod
def _notify_linux(title: str, message: str, urgency: str = "normal")
```

**Linux 通知实现**，使用 `notify-send` 命令：

```bash
notify-send --urgency={urgency} "{title}" "{message}"
```

- 依赖 `libnotify`（大多数桌面 Linux 发行版预装）
- `urgency` 直接传递给 `notify-send`，支持 `low` / `normal` / `critical`
- 在无桌面环境的服务器上 `notify-send` 不可用，此时静默失败

---

## 6. 紧急程度（Urgency）

通知紧急程度采用三级体系，与 Linux `notify-send` 标准对齐：

| 级别       | 含义     | 使用场景              | 平台支持          |
| ---------- | -------- | --------------------- | ----------------- |
| `low`      | 低优先级  | 日常信息提醒           | 仅 Linux 生效     |
| `normal`   | 普通     | 默认级别，状态变更等    | 仅 Linux 生效     |
| `critical` | 紧急     | 验收驳回等重要事件      | 仅 Linux 生效     |

> Windows 和 macOS 平台当前不传递 `urgency` 参数，所有通知以相同优先级展示。

## 7. 配置项

所有通知相关配置位于 `notification` 命名空间下，默认值定义在 `Config.DEFAULT_CONFIG` 中：

```python
"notification": {
    "enabled": False,           # 总开关，默认关闭
    "type": "toast",            # 通知类型（预留，当前仅支持 toast）
    "on_status_change": True,   # 状态变更时是否通知
    "on_submit": True,          # 任务提交时是否通知
    "on_verify": True,          # 验收结果是否通知
}
```

### 配置层级

```
notification.enabled = False         ← 总开关（最高优先级）
  │
  ├─ notification.on_status_change   ← 状态变更通知开关
  ├─ notification.on_submit          ← 任务提交通知开关
  └─ notification.on_verify          ← 验收结果通知开关
```

当 `enabled = False` 时，`notify()` 方法直接返回，三个子开关均不生效。只有 `enabled = True` 时，子开关才能独立控制各事件的通知行为。

## 8. 设计要点

### 8.1 跨平台抽象

统一的 `notify()` 入口根据 `sys.platform` 自动分发到对应平台实现，调用方无需关心平台差异。

### 8.2 零依赖策略

不依赖 `win10toast`、`plyer` 等第三方通知库。完全通过操作系统自带命令实现：

| 平台    | 命令/机制                                       |
| ------- | ----------------------------------------------- |
| Windows | PowerShell → WinRT `ToastNotificationManager`   |
| macOS   | `osascript` → AppleScript `display notification` |
| Linux   | `notify-send`（libnotify）                       |

### 8.3 优雅降级

- Windows 平台：WinRT API 失败 → 回退到 MessageBox
- 所有平台：命令执行异常 → 静默捕获，不抛出异常

### 8.4 非阻塞设计

- 通知失败不影响主流程（所有异常被 `except Exception` 捕获）
- 子进程设置 5 秒超时，防止命令挂起

### 8.5 配置驱动

- 总开关 + 事件级开关的两层控制
- 用户可精细控制哪些事件触发通知

### 8.6 可独立测试

- 不依赖 TaskManager、Database 或任何其他业务模块
- 仅依赖 `Config` 对象，可轻松 mock

### 9.2 已实现的集成点

`Notifier` 的三个事件方法已与 `TaskManager` 中的三个业务动作完成对接：

```
TaskManager.change_status()  →  Notifier.on_status_change()  ✅ 已集成
TaskManager.submit_task()    →  Notifier.on_submit()          ✅ 已集成
TaskManager.verify_task()    →  Notifier.on_verify()          ✅ 已集成
```

### 9.3 实际集成实现

采用 **构造注入 + 内部辅助方法** 的方式，将 Notifier 集成到 TaskManager 中。

#### 构造注入

`TaskManager.__init__` 接受可选的 `notifier` 关键字参数（默认 `None`），由 CLI `_init()` 负责创建并注入：

```python
class TaskManager:
    def __init__(self, db: Database, terminal_id: str, *, notifier=None):
        self.db = db
        self.terminal_id = terminal_id
        self.notifier = notifier
```

#### 辅助回调方法

TaskManager 内部通过三个 `_notify_*` 辅助方法封装通知调用，所有方法均以 `if self.notifier:` 守卫，确保未注入 Notifier 时不产生任何副作用：

```python
def _notify_status_change(self, task):
    if self.notifier:
        self.notifier.on_status_change(task.task_id, old_status, task.status.value, task.title)

def _notify_submit(self, task):
    if self.notifier:
        self.notifier.on_submit(task.task_id, task.title, self.terminal_id)

def _notify_verify(self, task, approved, comment):
    if self.notifier:
        self.notifier.on_verify(task.task_id, task.title, approved, comment)
```

#### 调用时机

- `change_status()` — 状态变更成功后调用 `_notify_status_change(task)`
- `submit_task()` — 提交成功后调用 `_notify_submit(task)`
- `verify_task()` — 验收完成后调用 `_notify_verify(task, approved, comment)`

### 9.4 其他集成建议

- **配置文件控制通知开关**：已实现（`notification.enabled`），集成后即可生效
- **通知频率限制**：可在 `notify()` 中添加节流逻辑（如同一 task_id 的通知间隔不少于 N 秒）
- **通知类型扩展**：`notification.type` 配置已预留，未来可支持 webhook、邮件等通知渠道
- **`__init__.py` 导出**：集成时应在 `__init__.py` 中导出 `Notifier` 类，方便外部导入

## 10. 使用示例

### 10.1 基本初始化与通知发送

```python
from cli_anything.utils.config import Config
from cli_anything.notification.notifier import Notifier

# 加载配置（需确保 notification.enabled = True）
config = Config()

# 创建通知器
notifier = Notifier(config)

# 检查是否启用
if notifier.enabled:
    # 发送普通通知
    notifier.notify("CLI-Anything", "任务处理完成")

    # 发送紧急通知（仅 Linux 有视觉差异）
    notifier.notify("⚠️ 警告", "磁盘空间不足", urgency="critical")
```

### 10.2 任务状态变更通知

```python
# 假设某任务从 "进行中" 变为 "待验收"
notifier.on_status_change(
    task_id="TASK-042",
    old_status="in_progress",
    new_status="pending_verify",
    title="实现用户登录功能"
)
# 桌面将弹出通知：
#   标题: 任务状态变更
#   正文: [TASK-042] 实现用户登录功能
#         in_progress → pending_verify
```

### 10.3 任务提交通知

```python
notifier.on_submit(
    task_id="TASK-042",
    title="实现用户登录功能",
    terminal_id="copilot-terminal-1"
)
# 桌面将弹出通知：
#   标题: 📤 任务已提交
#   正文: [TASK-042] 实现用户登录功能
#         来自终端: copilot-terminal-1
```

### 10.4 验收结果通知

```python
# 验收通过
notifier.on_verify(
    task_id="TASK-042",
```

### 10.7 单元测试示例

```python
from unittest.mock import MagicMock, patch
from cli_anything.notification.notifier import Notifier

def test_notify_disabled_by_default():

    config.get.side_effect = lambda key, default=None: {
        "notification.enabled": False,

    assert notifier.enabled is False

    # notify 应直接返回，不调用任何平台方法
    with patch.object(Notifier, '_notify_windows') as mock_win:
        notifier.notify("test", "message")
        mock_win.assert_not_called()


def test_on_verify_rejected_uses_critical_urgency():
    """验收驳回时应使用 critical 紧急级别"""
    config = MagicMock()
    config.get.side_effect = lambda key, default=None: {
        "notification.enabled": True,
        "notification.on_verify": True,
        "notification.type": "toast",
    }.get(key, default)

    notifier = Notifier(config)

    with patch.object(notifier, 'notify') as mock_notify:

        mock_notify.assert_called_once()
        _, kwargs = mock_notify.call_args
        assert kwargs.get('urgency') == 'critical'
```

## 11. 完整 API 速查

```python
class Notifier:
    def __init__(self, config: Config): ...

    @property
    def enabled(self) -> bool: ...

    # 核心方法
    def notify(self, title: str, message: str, urgency: str = "normal"): ...

    # 事件回调
    def on_status_change(self, task_id: str, old_status: str, new_status: str, title: str): ...
    def on_submit(self, task_id: str, title: str, terminal_id: str): ...
    def on_verify(self, task_id: str, title: str, approved: bool, comment: str): ...

    # 平台实现（静态方法，内部使用）
    @staticmethod
    def _notify_windows(title: str, message: str): ...
    @staticmethod
    def _notify_macos(title: str, message: str): ...
    @staticmethod
    def _notify_linux(title: str, message: str, urgency: str = "normal"): ...
```
