# 03 — 存储层（Storage Layer）

> 源文件：`src/cli_anything/storage/database.py`（~288 行）

---

## 1. 模块概述

CLI-Anything 的存储层基于 **SQLite** 实现，采用 **WAL（Write-Ahead Logging）** 日志模式以支持多终端并发访问。整个方案具有以下特点：

- **零部署**：无需安装额外数据库服务，一个 `.db` 文件即是完整数据库。
- **数据库文件默认路径**：`~/.cli-anything/tasks.db`。
- **自动建表**：首次连接时自动创建所有表和索引，开箱即用。
- **平滑迁移**：对已有数据库通过 `PRAGMA table_info` 检测并补充新增列，无需手动升级。

依赖的数据模型定义在 `cli_anything.core.models` 中（`Task`、`TaskLog`、`Terminal`）。

## 2. 数据库配置

连接建立时（`connect()` 方法）依次执行以下 PRAGMA 配置：

| PRAGMA | 值 | 说明 |
|---|---|---|
| `journal_mode` | `WAL` | 启用 Write-Ahead Logging，允许一写多读并发 |
| `foreign_keys` | `ON` | 启用外键约束，确保引用完整性与级联删除生效 |
| `busy_timeout` | `15000`（15 秒） | 遇到数据库锁时最多等待 15 秒，避免立即抛出 `SQLITE_BUSY` |

此外，连接时还设置了 `row_factory = sqlite3.Row`，使查询结果可通过列名（如 `row["status"]`）访问。每个线程使用独立 SQLite 连接，写操作通过实例级可重入写锁串行化，`close()` 会释放该实例创建过的所有线程连接。

```python
# 连接初始化代码片段
conn = sqlite3.connect(self.db_path, timeout=15.0, check_same_thread=False)
conn.row_factory = sqlite3.Row
conn.execute("PRAGMA journal_mode=WAL")
conn.execute("PRAGMA foreign_keys=ON")
conn.execute("PRAGMA busy_timeout=15000")
```

## 3. 表结构

存储层共维护 **3 张表** 和 **8 个索引**。

### 3.1 tasks 表（24 列）

任务主表，存储所有主任务和子任务。

```sql
CREATE TABLE IF NOT EXISTS tasks (
    id              TEXT PRIMARY KEY,
    title           TEXT NOT NULL,
    description     TEXT DEFAULT '',
    status          TEXT NOT NULL DEFAULT 'pending',
    task_type       TEXT NOT NULL DEFAULT 'master',
    priority        INTEGER DEFAULT 3 CHECK (priority BETWEEN 1 AND 5),
    tags            TEXT DEFAULT '[]',
    parent_id       TEXT REFERENCES tasks(id) ON DELETE CASCADE,
    created_by      TEXT NOT NULL DEFAULT '',
    claimed_by      TEXT DEFAULT NULL,
    claimed_at      TEXT DEFAULT NULL,
    submitted_at    TEXT DEFAULT NULL,
    verified_by     TEXT DEFAULT NULL,
    verified_at     TEXT DEFAULT NULL,
    verify_comment  TEXT DEFAULT '',
    test_status     TEXT DEFAULT 'not_run',
    test_report     TEXT DEFAULT '{}',
    test_path       TEXT DEFAULT '',
    work_dir        TEXT DEFAULT '',
    reviewer        TEXT DEFAULT NULL,
    review_status   TEXT DEFAULT 'not_required',
    review_comment  TEXT DEFAULT '',
    created_at      TEXT NOT NULL DEFAULT (datetime('now', 'localtime')),
    updated_at      TEXT NOT NULL DEFAULT (datetime('now', 'localtime'))
);
```

## 4. Database 类

`Database` 类是存储层的唯一入口，封装了连接管理和所有 CRUD 操作。

### 4.1 连接管理

#### `__init__(db_path: str = DEFAULT_DB_PATH)`

构造函数，接收数据库文件路径，默认为 `~/.cli-anything/tasks.db`。此时 **不会** 立即建立连接。

#### `connect() → sqlite3.Connection`

建立数据库连接，执行以下步骤：

1. 如已有连接则直接返回（幂等）。
2. 确保数据库目录存在（`os.makedirs(..., exist_ok=True)`）。
3. 调用 `sqlite3.connect()` 创建连接（timeout=5.0 秒）。
4. 设置 `row_factory = sqlite3.Row`。
5. 执行三条 PRAGMA（WAL / foreign_keys / busy_timeout）。
6. 执行 `_SCHEMA_SQL` 建表脚本（`executescript`）。
7. 调用 `_migrate_review_columns()` 进行兼容性迁移。

#### `close()`

关闭连接并将内部 `_conn` 置为 `None`。

#### `conn` 属性（懒初始化）

```python
@property
def conn(self) -> sqlite3.Connection:
    if self._conn is None:
        return self.connect()
    return self._conn
```

首次访问时自动调用 `connect()`，后续直接返回已有连接。所有 CRUD 方法内部均通过 `self.conn` 获取连接，因此使用者无需显式调用 `connect()`。

---

### 4.2 迁移

#### `_migrate_review_columns()`

为兼容旧版本数据库，检测 `tasks` 表是否存在以下三列：

| 列名 | 定义 |
|---|---|
| `reviewer` | `TEXT DEFAULT NULL` |
| `review_status` | `TEXT DEFAULT 'not_required'` |
| `review_comment` | `TEXT DEFAULT ''` |

**实现方式**：

1. 通过 `PRAGMA table_info(tasks)` 获取现有列名集合。
2. 遍历待迁移列，若不存在则执行 `ALTER TABLE tasks ADD COLUMN ...`。
3. 提交事务。

这种基于 **列存在性检测** 的迁移策略（而非版本号）简单可靠，保证旧数据库可以平滑升级到新版本。

### 4.3 Task CRUD

#### `insert_task(task: Task) → Task`

插入新任务。将 Task 对象序列化为字典，动态构建列名和占位符：

```python
d = task.to_dict()
cols = ", ".join(d.keys())
placeholders = ", ".join(["?"] * len(d))
# INSERT INTO tasks (col1, col2, ...) VALUES (?, ?, ...)
```

#### `get_task(task_id: str) → Optional[Task]`

按 ID 查询单个任务，返回 `Task` 对象或 `None`。

```sql
SELECT * FROM tasks WHERE id = ?
```

#### `list_tasks(status, task_type, parent_id, claimed_by, tag, limit) → list[Task]`

支持多条件过滤的任务列表查询。采用 **动态 SQL 构建** 方式：

- 根据非空参数动态拼接 `WHERE` 子句。
- 所有参数通过 `?` 占位符传入，**防止 SQL 注入**。
- `tag` 过滤使用 `LIKE '%"tag_name"%'` 模糊匹配 JSON 数组中的标签。
- 默认排序：`ORDER BY priority ASC, created_at DESC`（高优先级靠前，同优先级按时间倒序）。
- 默认 `limit = 100`。

```python
# 动态构建示例
conditions: list[str] = []
params: list = []
if status:
    conditions.append("status = ?")
    params.append(status)
# ... 其他条件 ...
where = " AND ".join(conditions) if conditions else "1=1"
```

#### `update_task(task: Task) → Task`

全字段更新任务。自动将 `updated_at` 设为当前时间：

```python
task.updated_at = datetime.now().isoformat(timespec="seconds")
d = task.to_dict()
task_id = d.pop("id")
set_clause = ", ".join(f"{k} = ?" for k in d.keys())
# UPDATE tasks SET col1=?, col2=?, ... WHERE id = ?
```

#### `delete_task(task_id: str) → bool`

删除指定任务，返回是否删除成功（`rowcount > 0`）。由于外键设置了 `ON DELETE CASCADE`，删除父任务时会 **自动级联删除** 所有子任务及其关联日志。

```sql
DELETE FROM tasks WHERE id = ?
```

#### `count_subtasks_by_status(parent_id: str) → dict[str, int]`

统计指定父任务下各状态的子任务数量，返回如 `{"pending": 3, "done": 2}` 的字典。

```sql
SELECT status, COUNT(*) as cnt FROM tasks WHERE parent_id = ? GROUP BY status
```

### 4.4 TaskLog CRUD

#### `insert_log(log: TaskLog) → TaskLog`

插入操作日志，与 `insert_task` 类似采用动态列构建。插入后将 `lastrowid` 回写到 `log.id`，使调用方可获取自增 ID。

```python
cur = self.conn.execute(f"INSERT INTO task_logs ({cols}) VALUES ({placeholders})", ...)
log.id = cur.lastrowid
```

#### `list_logs(task_id, action, limit) → list[TaskLog]`

查询操作日志，支持按 `task_id` 和 `action` 过滤。

- 排序：`ORDER BY timestamp DESC`（最新在前）。
- 默认 `limit = 50`。
- 同样采用动态 WHERE 构建 + 参数化查询。

---

### 4.5 Terminal CRUD

#### `upsert_terminal(terminal: Terminal) → Terminal`

使用 SQLite 的 `INSERT ... ON CONFLICT DO UPDATE` 语法实现 **UPSERT**（幂等注册）：

```sql
INSERT INTO terminals (id, name, role, ...) VALUES (?, ?, ?, ...)
ON CONFLICT(id) DO UPDATE SET name=excluded.name, role=excluded.role, ...
```

- 终端首次注册时执行 INSERT。
- 已存在时更新除 `id` 之外的所有字段。
- 保证无论调用多少次，结果一致。

#### `get_terminal(terminal_id: str) → Optional[Terminal]`

按 ID 查询单个终端，返回 `Terminal` 对象或 `None`。

#### `list_terminals() → list[Terminal]`

返回所有已注册终端，按 `last_active DESC` 排序（最近活跃的在前）。

## 5. 设计要点

### 5.1 WAL 模式

SQLite 的 WAL 模式将写入操作记录到独立的日志文件（`tasks.db-wal`），读操作继续访问主文件。这使得 **多终端可以同时读取，一个终端可以同时写入**，显著提升并发性能。

### 5.2 外键级联删除

`parent_id` 和 `task_id` 均设置了 `ON DELETE CASCADE`：

- 删除一个主任务 → 自动删除其所有子任务。
- 删除一个任务 → 自动删除其所有操作日志。

这避免了手动清理孤儿记录的复杂性。

### 5.3 Row Factory

连接时设置 `sqlite3.Row` 作为 row_factory，使查询结果同时支持 **索引访问**（`row[0]`）和 **列名访问**（`row["status"]`），代码可读性更佳。

### 5.4 动态 SQL 构建

`list_tasks` 和 `list_logs` 方法根据传入参数动态拼接 `WHERE` 子句：

- 仅拼接非空条件，无多余过滤。
- 始终使用 `?` 占位符 + 参数列表，**杜绝 SQL 注入**。
- 无条件时退化为 `WHERE 1=1`（全量查询）。

### 5.5 迁移兼容

不使用版本号表，而是通过 `PRAGMA table_info` 直接检测列是否存在。优势：

- 逻辑简单，无需维护迁移版本状态。
- 幂等：多次运行不会重复添加列。
- 适合单表少量列变更的轻量级场景。

### 5.6 自动时间戳

`created_at` 和 `updated_at` 的默认值使用 SQLite 内置函数：

```sql
DEFAULT (datetime('now', 'localtime'))
```

- 插入时自动填充为本地时间。
- `update_task` 方法在 Python 层面手动更新 `updated_at`，确保每次修改都能刷新时间戳。

### 5.7 UPSERT（幂等注册）

`terminals` 表使用 `INSERT ... ON CONFLICT(id) DO UPDATE SET ...`：

- 新终端：执行 INSERT。
- 已有终端：更新字段（如 `last_active`、`pid` 等）。
- 调用方无需判断终端是否已存在，简化业务逻辑。

## 6. 性能考量

| 措施 | 效果 |
|---|---|
| **5 个 tasks 索引** | 覆盖 status、task_type、parent_id、claimed_by、priority 常见查询路径 |
| **3 个 task_logs 索引** | 覆盖 task_id、action、timestamp 查询和排序 |
| **busy_timeout=5000** | 避免并发时立即抛出 `SQLITE_BUSY`，最多等待 5 秒 |
| **WAL 模式** | 支持一写多读并发，读操作不阻塞写操作 |
| **sqlite3.connect(timeout=5.0)** | 连接级别的超时保护 |
| **单文件零部署** | 无需数据库服务，适合本地开发与 CLI 工具场景 |
| **懒初始化连接** | 仅在首次使用时建立连接，避免不必要的 I/O 开销 |

## 7. 使用示例

### 7.1 基础用法

```python
from cli_anything.storage.database import Database

# 使用默认路径（~/.cli-anything/tasks.db）
db = Database()

# 也可指定自定义路径
db = Database(db_path="D:/projects/my-app/tasks.db")

# 无需显式 connect()，首次操作时自动连接
tasks = db.list_tasks(status="pending", limit=10)

# 查询单个任务
task = db.get_task("task-001")

# 关闭连接
db.close()
```

### 7.2 多终端协同场景（Copilot + Qwen + Gemini）

本项目的典型使用环境为三个 AI CLI 终端同时连接同一个 SQLite 数据库进行协同开发。存储层的 WAL 模式正是为此设计。

```
┌────────────────┐  ┌────────────────┐  ┌────────────────┐
│  Copilot CLI   │  │   Qwen CLI     │  │  Gemini CLI    │
│  role: master  │  │  role: worker  │  │  role: worker  │
└───────┬────────┘  └───────┬────────┘  └───────┬────────┘
        │                   │                   │
        └───────────────────┼───────────────────┘
                            ▼
                  ~/.cli-anything/tasks.db  (WAL 模式)
```

#### 终端注册（三终端各自执行）

每个终端启动时通过 `upsert_terminal` 注册自身信息。UPSERT 语义保证重复启动不会产生冲突。

```python
from cli_anything.core.models import Terminal, TerminalRole

# --- Copilot 终端（主终端） ---
copilot = Terminal(
    id="copilot-main",
    name="Copilot-Master",
    role=TerminalRole.MASTER,
    type="powershell",
    pid=12345,
)
db.upsert_terminal(copilot)

# --- Qwen 终端（子终端） ---
qwen = Terminal(
    id="qwen-worker-01",
    name="Qwen-Worker",
    role=TerminalRole.WORKER,
    type="powershell",
    pid=23456,
)
db.upsert_terminal(qwen)

# --- Gemini 终端（子终端） ---
gemini = Terminal(
    id="gemini-worker-02",
    name="Gemini-Worker",
    role=TerminalRole.WORKER,
    type="cmd",
    pid=34567,
)
db.upsert_terminal(gemini)

# 查看所有已注册终端
for t in db.list_terminals():
    print(f"{t.name} ({t.role.value}) — PID {t.pid}")
```

### 7.3 并发安全说明

在上述场景中，Copilot、Qwen、Gemini 三个进程同时访问 `~/.cli-anything/tasks.db`：

| 操作 | 并发表现 |
|---|---|
| 多终端同时读取任务列表 | ✅ WAL 模式下读读不冲突 |
| 一个终端写入 + 其他终端读取 | ✅ WAL 模式下读写不冲突 |
| 两个终端同时写入 | ⏳ `busy_timeout=5000` 排队等待，5 秒内通常可获锁 |
| 极端写入竞争 | ❌ 超时后抛出 `sqlite3.OperationalError: database is locked`，上层需重试 |
