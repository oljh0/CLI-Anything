## 2. 安装方式

### 2.1 从源码安装

```bash
# 克隆仓库
git clone <repo-url>
cd CLI-Anything

# 安装（仅核心依赖）
pip install -e .

# 安装（含开发/测试依赖）
pip install -e ".[dev]"
```

> **提示**：`-e`（editable）模式允许你在修改源码后无需重新安装即可生效，适合开发阶段使用。

### 2.2 依赖说明

| `typer` | `>=0.9.0` | CLI 框架，提供命令行参数解析、子命令路由、帮助文档生成 |

### 2.3 入口点

安装完成后，系统注册一个 CLI 命令：

```
cli-anything = "cli_anything.cli:app"
```

- **命令名**：`cli-anything`
- **入口模块**：`src/cli_anything/cli.py` 中的 `app`（Typer 实例）
- **调用方式**：在任意终端中输入 `cli-anything <子命令>` 即可执行

```bash
# 验证安装是否成功
cli-anything --help
```

## 3. 配置文件

### 3.1 配置文件路径

| 项目 | 路径 |
|------|------|
| 默认配置文件 | `~/.cli-anything/config.yaml` |
| 示例配置文件 | 项目根目录 `config/config.example.yaml` |

- 执行 `cli-anything init` 时自动创建 `~/.cli-anything/` 目录
- 如果配置文件不存在，初始化时会尝试从项目内 `config/config.example.yaml` 复制；若找不到示例文件则使用代码内嵌的默认配置
- 配置文件采用 YAML 格式，UTF-8 编码

### 3.2 完整配置项说明

以下逐段解释 `config.example.yaml` 中的每个配置块及其默认值。

#### 3.2.1 `database` — 数据库配置

```yaml
database:
  path: "~/.cli-anything/tasks.db"   # 数据库文件路径，支持 ~ 展开
  wal_mode: true                      # 是否启用 WAL (Write-Ahead Logging) 模式
  busy_timeout: 5000                  # 数据库忙等待超时（毫秒）
```

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `path` | `string` | `~/.cli-anything/tasks.db` | SQLite 数据库文件路径。支持 `~` 前缀自动展开为用户主目录。 |
| `wal_mode` | `bool` | `true` | 启用 WAL 模式后，多终端可以并发读取而不互相阻塞。强烈建议保持 `true`。 |
| `busy_timeout` | `int` | `5000` | 当数据库被锁定时等待的毫秒数。多终端并发写入时防止 `database is locked` 错误。 |

#### 3.2.2 `terminal` — 终端身份配置

```yaml
terminal:
  role: "master"        # 终端角色：master（主终端）或 worker（子终端）
  name: ""              # 终端显示名称，留空由用户在 init 时指定
  auto_detect: true     # 是否自动检测终端类型（如 PowerShell/Bash/WSL 等）
  id: ""                # 终端唯一 ID，留空则自动生成 UUID
```

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `role` | `string` | `"worker"` | 终端角色。`master` 拥有创建、拆解、验收权限；`worker` 拥有领取、开始、提交权限。代码内嵌默认为 `worker`，示例配置为 `master`。 |
| `name` | `string` | `""` | 终端的人类可读名称，如 `"主控台"`、`"开发-1"`。在任务列表和日志中标识终端来源。 |
| `auto_detect` | `bool` | `true` | 自动检测当前终端环境类型（PowerShell、CMD、Bash、Zsh、WSL 等）。 |
| `id` | `string` | `""` | 终端唯一标识符。留空时在注册阶段自动生成 UUID。手动指定可用于固定终端身份。 |

#### 3.2.3 `mcp_server` — MCP Server 配置

```yaml
mcp_server:
  transport: "stdio"          # 传输方式：目前仅支持 stdio
  sse_port: 8000              # SSE 传输端口（预留，当前未启用）
  sse_host: "127.0.0.1"       # SSE 传输绑定地址（预留，当前未启用）
```

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `transport` | `string` | `"stdio"` | MCP 传输方式。当前仅实现 `stdio`（标准输入/输出），AI Agent 通过子进程与 MCP Server 通信。 |
| `sse_port` | `int` | `8000` | SSE（Server-Sent Events）传输端口。**预留配置**，当前版本未实现 SSE 传输。 |
| `sse_host` | `string` | `"127.0.0.1"` | SSE 传输绑定地址。**预留配置**，当前版本未实现 SSE 传输。 |

#### 3.2.4 `dashboard` — Web 看板配置

```yaml
dashboard:
  port: 8080                 # Web Dashboard 监听端口
  host: "127.0.0.1"          # 绑定地址（127.0.0.1 仅本机，0.0.0.0 允许局域网访问）
  auto_open: true            # 启动后是否自动打开浏览器
  refresh_interval: 5        # 前端自动刷新间隔（秒）
```

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `port` | `int` | `8080` | Dashboard HTTP 服务端口。确保端口未被占用。 |
| `host` | `string` | `"127.0.0.1"` | 绑定地址。默认仅本机访问；改为 `"0.0.0.0"` 可允许局域网其他设备访问。 |
| `auto_open` | `bool` | `true` | 启动 Dashboard 后是否自动在默认浏览器中打开页面。 |
| `refresh_interval` | `int` | `5` | 前端页面自动刷新数据的时间间隔（秒）。WebSocket 提供实时推送，此项为降级轮询方案。 |

#### 3.2.5 `testing` — 测试运行配置

```yaml
testing:
  runner: "pytest"                    # 测试运行器
  default_args: ["-v", "--tb=short"]  # 默认测试参数
  timeout: 300                        # 测试超时时间（秒）
  auto_run_on_submit: true            # 提交任务时是否自动运行测试
  test_dir: "tests/"                  # 测试目录
```

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `runner` | `string` | `"pytest"` | 测试运行器名称。当前仅支持 `pytest`。 |
| `default_args` | `list[string]` | `["-v", "--tb=short"]` | 传递给 pytest 的默认参数。`-v` 详细输出，`--tb=short` 简短回溯。 |
| `timeout` | `int` | `300` | 单次测试运行的超时时间（秒），超时后强制终止。 |
| `auto_run_on_submit` | `bool` | `true` | Worker 执行 `submit` 命令时是否自动触发测试。设为 `false` 则仅提交不测试。 |
| `test_dir` | `string` | `"tests/"` | 测试文件所在目录，相对于项目根目录。 |

#### 3.2.6 `display` — 显示配置

```yaml
display:
  color: true                    # 是否启用彩色输出
  table_format: "rich"           # 表格渲染引擎
  date_format: "%Y-%m-%d %H:%M"  # 日期时间显示格式
```

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `color` | `bool` | `true` | 终端输出是否启用 Rich 彩色渲染。CI 环境或重定向输出时可设为 `false`。 |
| `table_format` | `string` | `"rich"` | 表格渲染方式。当前使用 `rich` 库的表格组件。 |
| `date_format` | `string` | `"%Y-%m-%d %H:%M"` | 日期时间的格式化字符串，遵循 Python `strftime` 规范。 |

#### 3.2.7 `notification` — 通知配置

```yaml
notification:
  enabled: false            # 是否启用通知
  type: "toast"             # 通知类型：toast（系统通知）
  on_status_change: true    # 任务状态变更时通知
  on_submit: true           # 任务提交时通知
  on_verify: true           # 任务验收时通知
```

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `enabled` | `bool` | `false` | 通知总开关。默认关闭。 |
| `type` | `string` | `"toast"` | 通知类型。`toast` 表示操作系统原生桌面通知。 |
| `on_status_change` | `bool` | `true` | 任务状态流转时是否发送通知。 |
| `on_submit` | `bool` | `true` | Worker 提交任务时是否通知 Master。 |
| `on_verify` | `bool` | `true` | Master 验收任务时是否通知 Worker。 |

> ✅ **已集成**：通知模块已通过构造注入集成到 `TaskManager`。CLI 启动时自动创建 `Notifier` 实例并注入 `TaskManager`，状态变更 / 任务提交 / 验收结果会自动触发桌面通知。将 `enabled` 设为 `true` 即可启用。

#### 3.2.8 `aliases` — 命令别名

```yaml
aliases:
  todo: "available"
  wip: "my --status in_progress"
  done: "my --status done"
```

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `todo` | `string` | `"available"` | 输入 `cli-anything todo` 等同于 `cli-anything available` |
| `wip` | `string` | `"my --status in_progress"` | 快速查看自己正在进行的任务 |
| `done` | `string` | `"my --status done"` | 快速查看自己已完成的任务 |

> ✅ **已实现**：CLI 入口函数 `main()` 在 Typer 解析参数前调用 `_resolve_aliases()`，自动将用户输入的别名替换为实际命令。支持带参数的别名展开（如 `wip` → `my --status in_progress`）。

### 3.3 配置加载优先级

配置系统采用分层覆盖机制，优先级从高到低：

```
命令行参数  >  用户配置文件  >  代码内嵌默认值
        ↑              ↑              ↑
   最高优先级       中等优先级       最低优先级
```

1. **命令行参数** — 直接通过 `--role`、`--name`、`--host`、`--port` 等参数传入，优先级最高
2. **用户配置文件** — `~/.cli-anything/config.yaml` 中的用户自定义配置
3. **代码内嵌默认值** — `config.py` 中 `_DEFAULT_CONFIG` 字典定义的硬编码默认值

### 3.4 深度合并机制

配置加载时使用 `_deep_merge(base, override)` 函数进行递归合并：

```python
def _deep_merge(base: dict, override: dict) -> dict:
    """深度合并两个字典，override 覆盖 base"""
    result = base.copy()
    for key, value in override.items():
        if key in result and isinstance(result[key], dict) and isinstance(value, dict):
            result[key] = _deep_merge(result[key], value)
        else:
            result[key] = value
    return result
```

**合并规则**：

| 场景 | 行为 |
|------|------|
| 两侧都是字典 | 递归合并，逐个 key 处理 |
| override 侧非字典 | 直接替换整个值 |
| 列表类型 | **直接替换**（不会追加合并） |
| override 中有新 key | 直接添加到结果中 |

**示例**：

```yaml
# 代码默认值
testing:
  runner: "pytest"
  default_args: ["-v", "--tb=short"]
  timeout: 300

# 用户配置（仅修改 timeout）
testing:
  timeout: 600

# 合并结果（其余字段保留默认值）
testing:
  runner: "pytest"
  default_args: ["-v", "--tb=short"]
  timeout: 600    # ← 被用户值覆盖
```

> **注意**：如果用户配置中修改了 `default_args`，整个列表会被替换而非追加。


## 5. 快速入门

### 5.1 初始化

在不同终端窗口中分别初始化为 Master 和 Worker：

```bash
# 终端 1 — 初始化为主终端
cli-anything init --role master --name "主控台"

# 终端 2 — 初始化为子终端
cli-anything init --role worker --name "开发-1"

# 终端 3 — 初始化为另一个子终端
cli-anything init --role worker --name "开发-2"
```

`init` 命令会：

1. 创建 `~/.cli-anything/` 目录
2. 生成 `config.yaml`（从示例文件复制或使用默认值）
3. 设置终端角色和名称
4. 初始化 SQLite 数据库

### 5.2 创建任务（Master）

```bash
# 创建主任务
cli-anything create "实现用户认证" -d "JWT + bcrypt 方案" -p 1 -t "auth,security"

# 拆解为子任务
cli-anything decompose <task-id> '[{"title":"登录接口"},{"title":"注册接口"},{"title":"Token刷新"}]'

# 查看任务详情及子任务
cli-anything show <task-id>
```

### 5.3 领取并完成任务（Worker）

```bash
# 查看可领取的任务
cli-anything available

# 领取任务
cli-anything claim <subtask-id>

# 开始工作（状态变为 in_progress）
cli-anything start <subtask-id>

# 完成后提交（自动运行测试）
cli-anything submit <subtask-id>
```

### 5.4 审核验收（Master）

```bash
# 通过验收
cli-anything verify <subtask-id> --approve -c "LGTM，代码质量好"

# 驳回
cli-anything verify <subtask-id> --reject -c "缺少边界测试用例"
```

### 5.5 完整工作流示例

以下展示一个典型的 Master/Worker 协作流程：

```
Master 终端                              Worker 终端
─────────────                            ─────────────
1. init --role master                    1. init --role worker
2. create "用户认证模块"                     │
3. decompose <id> '[...]'                  │
4. (等待 Worker 领取)                      2. available
   │                                     3. claim <subtask-id>
   │                                     4. start <subtask-id>
   │                                     5. (编写代码...)
   │                                     6. submit <subtask-id>
5. list --status submitted                  │  ← 自动运行 pytest
6. verify <id> --approve                    │
   │                                     7. my --status done ✓
7. progress <parent-id>
   → 1/3 完成，进度 33%
```

**任务状态流转**：

```
pending → claimed → in_progress → submitted → done
                                           → rejected → in_progress（重新开始）
```

## 6. MCP Server 部署

MCP（Model Context Protocol）Server 允许 AI Agent（如 GitHub Copilot CLI、Claude Desktop）通过标准化协议调用 CLI-Anything 的 10 个工具。

### 6.1 启动 MCP Server

```bash
cli-anything serve
```

默认使用 `stdio` 传输方式 — 通过标准输入/输出与 AI Agent 通信。

### 6.2 提供的 MCP 工具

| 工具名 | 说明 |
|--------|------|
| `task_create` | 创建新任务 |
| `task_decompose` | 拆解任务为子任务 |
| `task_list` | 列出任务（支持过滤） |
| `task_claim` | 领取任务 |
| `task_submit` | 提交任务 |
| `task_verify` | 验收任务 |
| `task_progress` | 查看任务进度 |
| `task_update` | 更新任务属性 |
| `task_delete` | 删除任务 |
| `task_log` | 查看操作日志 |

### 6.3 Claude Desktop 配置

编辑 Claude Desktop 配置文件：

### 6.5 传输方式说明

当前版本仅支持 `stdio` 传输：

```yaml
mcp_server:
  transport: "stdio"    # 唯一支持的传输方式
```

- **stdio**：AI Agent 将 `cli-anything serve` 作为子进程启动，通过 stdin/stdout 通信
- **SSE**：配置文件中有 `sse_port` / `sse_host` 预留字段，但当前版本 **未实现** SSE 传输

### 7.4 生产环境建议

CLI-Anything 的 Dashboard 主要面向本地开发和团队内网场景。如需在更正式的环境中部署，请注意以下事项：

| 关注点 | 建议 |
|--------|------|
| **反向代理** | 使用 Nginx/Caddy 做反向代理，提供 HTTPS 和域名访问 |
| **进程管理** | 使用 systemd / PM2 / supervisor 守护进程，保证崩溃后自动重启 |
| **访问控制** | Dashboard 无内建认证，请通过反向代理层添加 Basic Auth 或 OAuth |
| **数据库** | SQLite 单文件适合少量并发；高并发场景请评估数据库锁竞争情况 |
| **日志** | 建议将 uvicorn 日志重定向到文件，方便排查问题 |

**Nginx 反向代理示例**：

```nginx
server {
    listen 443 ssl;
    server_name tasks.example.com;

    ssl_certificate     /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
    }
}
```

> **注意**：WebSocket 连接需要 `proxy_set_header Upgrade` 和 `Connection "upgrade"` 配置。

## 8. 多终端协作

CLI-Anything 的核心设计就是多终端协作。所有终端共享同一个 SQLite 数据库，通过 WAL 模式实现并发安全。

### 8.1 同一台机器 — 多终端窗口

最常见的使用场景：在同一台机器上打开多个终端窗口。

```
┌──────────────────────────────────────────────┐
│               同一台机器                       │
│                                              │
│  ┌─────────────┐  ┌─────────────┐            │
│  │ 终端 1       │  │ 终端 2       │            │
│  │ role: master │  │ role: worker │            │
│  │ name: 主控台  │  │ name: 开发-1 │            │
│  └──────┬───────┘  └──────┬──────┘            │
│         │                 │                   │
│         └────────┬────────┘                   │
│                  ▼                            │
│     ~/.cli-anything/tasks.db                  │
│           (SQLite WAL)                        │
└──────────────────────────────────────────────┘
```

**配置要点**：

- 所有终端共享同一个 `~/.cli-anything/tasks.db`
- 每个终端通过 `init` 设置不同的 `role` 和 `name`
- 终端 ID 自动生成，用于标识任务归属

### 8.2 同一网络 — 多设备

多台设备通过 Web Dashboard 共享视图，但**每台设备需独立运行 CLI**：

```
┌────────────┐       ┌────────────┐
│ 设备 A      │       │ 设备 B      │
│ Master CLI  │       │ Worker CLI  │
│ tasks.db    │       │ tasks.db    │
│ Dashboard   │←─────→│ (浏览器)     │
│ :8080       │ HTTP  │             │
└────────────┘       └────────────┘
```

**方案说明**：

- 在 Master 设备上启动 Dashboard（`--host 0.0.0.0`），其他设备通过浏览器查看看板
- **重要限制**：当前版本各设备使用独立的本地 SQLite 数据库，不支持跨设备数据库同步
- 如需真正的跨设备协作，可考虑将 `database.path` 指向共享网络存储（NFS/SMB），但需注意 SQLite 在网络文件系统上的兼容性问题

### 8.3 WSL + Windows 混合环境

Windows 用户可以在 WSL 和原生 Windows 终端之间协作：

```bash
# Windows PowerShell（Master）
cli-anything init --role master --name "Win-主控台"

# WSL Bash（Worker）
cli-anything init --role worker --name "WSL-开发"
```

**注意事项**：

- WSL 和 Windows 的用户主目录不同（`/home/<user>` vs `C:\Users\<user>`）
- 需确保两个环境的 `database.path` 指向同一个文件
- 可在配置中使用 Windows 路径的 WSL 挂载点，如 `/mnt/c/Users/<user>/.cli-anything/tasks.db`
- 或者在 Windows 侧将路径设为 WSL 可访问的位置

**WSL 侧配置示例**：

```yaml
database:
  path: "/mnt/c/Users/YourName/.cli-anything/tasks.db"
```

## 9. 故障排查

### 9.1 数据库锁定（`database is locked`）

**症状**：多终端同时操作时出现 `sqlite3.OperationalError: database is locked`。

**排查步骤**：

1. 确认 WAL 模式已启用：

```yaml
database:
  wal_mode: true
```

2. 增大 `busy_timeout`（默认 5000ms）：

```yaml
database:
  busy_timeout: 10000  # 10 秒
```

3. 检查是否有僵尸进程持有锁：

```bash
# Linux/macOS
lsof ~/.cli-anything/tasks.db

# Windows
handle.exe tasks.db   # 需要 Sysinternals 工具
```

4. 确认数据库文件不在网络文件系统上（NFS/SMB 不支持 SQLite 锁机制）。

### 9.2 终端状态过期

**症状**：`terminals` 列表显示已关闭的终端仍为 active 状态。

**解决方案**：

```bash
# 运行健康检查，自动清理过期终端
cli-anything health --cleanup
```

健康检查器会根据心跳时间判断终端是否存活，并自动清理超时的终端注册信息。

### 9.3 配置不生效

**排查清单**：

1. **路径是否正确**：确认编辑的是 `~/.cli-anything/config.yaml` 而非项目内的 `config.example.yaml`
2. **YAML 格式是否正确**：

```bash
# 验证 YAML 语法
python -c "import yaml; yaml.safe_load(open('~/.cli-anything/config.yaml'.replace('~', __import__('os').path.expanduser('~'))))"
```

3. **缩进是否正确**：YAML 使用空格缩进，不支持 Tab
4. **配置项名称是否正确**：对照本文档第 3.2 节的字段名
5. **是否需要重启**：修改配置后需重新执行命令才能生效（Dashboard 和 MCP Server 需重启进程）

### 9.4 安装后 `cli-anything` 命令找不到

**排查步骤**：

1. 确认安装成功：`pip show cli-anything`
2. 确认 pip scripts 目录在 PATH 中：

```bash
python -m site --user-base
# 将输出路径下的 bin/ (Linux/macOS) 或 Scripts/ (Windows) 加入 PATH
```

3. 使用模块方式运行作为替代：

```bash
python -m cli_anything.cli --help
```

### 9.5 Dashboard 无法访问

**排查清单**：

1. 确认端口未被占用：

```bash
# Linux/macOS
lsof -i :8080

# Windows PowerShell
Get-NetTCPConnection -LocalPort 8080
```

2. 局域网访问时确认 `host` 设置为 `0.0.0.0`
3. 检查防火墙规则是否放行对应端口

## 10. 已知限制

以下是当前版本（0.1.0）的已知限制和未完成功能：

| 编号 | 限制 | 说明 |
|------|------|------|
| 1 | **MCP 仅支持 stdio 传输** | `mcp_server.transport` 当前仅实现 `stdio` 方式。SSE 传输的端口和地址配置已预留但未实现。 |
| 2 | **单文件 SQLite** | 使用单文件 SQLite 数据库，不适合高并发生产环境。WAL 模式 + busy_timeout 可缓解但无法根本解决高并发写入冲突。 |
| 3 | **无内建认证** | Dashboard 和 MCP Server 均无用户认证机制，依赖网络层（防火墙/反向代理）进行访问控制。 |
| 4 | **跨设备数据同步** | 各设备使用独立本地数据库，不支持自动跨设备数据同步。共享需依赖外部网络文件系统。 |
