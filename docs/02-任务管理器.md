# TaskManager 核心模块文档

> **源文件**：`src/cli_anything/core/task_manager.py`（417 行）
> **依赖模型**：`src/cli_anything/core/models.py`（Task / TaskLog / Terminal / 枚举 / VALID_TRANSITIONS）
> **存储层**：`src/cli_anything/storage/database.py`（Database）

---

## 1. 模块概述

`TaskManager` 是 CLI-Anything 系统的 **核心业务逻辑层**，承担任务全生命周期管理职责。

系统的 4 个访问层——CLI、MCP（Model Context Protocol）、Web、TUI——全部通过 `TaskManager` 执行业务操作，而不直接操作数据库。这使得业务规则（状态校验、日志记录、权限检查等）在所有入口处保持一致。

### 设计定位

```
┌──────────────────────────────────────┐
│  访问层: CLI / MCP / Web / TUI       │
├──────────────────────────────────────┤
│  TaskManager  ← 本模块（纯业务逻辑）  │
├──────────────────────────────────────┤
│  Database     ← 存储层（SQLite）      │
└──────────────────────────────────────┘
```

**核心原则**：

- 纯业务逻辑层，**不依赖任何 IO 框架**（HTTP、终端 UI 等）
- 所有写操作均记录 `TaskLog`，提供完整操作审计
- `terminal_id` 贯穿所有操作，用于追溯"谁做了什么"
- 状态变更严格遵循 `VALID_TRANSITIONS` 状态机
- 所有业务异常统一抛出 `TaskManagerError`

## 2. 类结构

### 2.1 TaskManagerError

```python
class TaskManagerError(Exception):
    """TaskManager 业务异常"""
```

自定义异常类，所有业务逻辑错误均通过此异常抛出。访问层可统一捕获该异常并转化为用户友好的错误提示（如 HTTP 400、CLI 错误消息等）。

**常见抛出场景**：

| 场景 | 错误消息示例 |
|------|------------|
| 标题为空 | `"任务标题不能为空"` |
| 优先级越界 | `"优先级必须在 1-5 之间"` |
| 父任务不存在 | `"父任务 {parent_id} 不存在"` |
| 任务不存在 | `"任务 {task_id} 不存在"` |
| 状态不允许转换 | `"不能从 X 转为 Y。允许的状态: ..."` |
| 非法领取 | `"只能领取 pending 状态的任务，当前状态: X"` |
| 释放他人任务 | `"只能释放自己领取的任务"` |

### 2.2 TaskManager

```python
class TaskManager:
    def __init__(self, db: Database, terminal_id: str = ""):
        self.db = db
        self.terminal_id = terminal_id
```

**构造参数**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `db` | `Database` | 数据库实例，提供持久化能力 |
| `terminal_id` | `str` | 当前操作终端的唯一标识，默认空字符串。所有写操作会自动关联此 ID |

## 3. 前置知识：枚举与状态机

理解 TaskManager 的行为需要先了解 `models.py` 中定义的枚举和状态流转规则。

### 3.1 TaskStatus — 任务状态

| 值 | 含义 | 说明 |
|----|------|------|
| `draft` | 草稿 | 需要审阅者审批后才能进入流程 |
| `pending` | 待领取 | 可被任意终端领取 |
| `claimed` | 已领取 | 已被某终端认领，尚未开始 |
| `in_progress` | 进行中 | 正在执行 |
| `submitted` | 已提交 | 等待主终端验收 |
| `done` | 完成 | **终态**，不可再变更 |
| `rejected` | 驳回 | 验收未通过，可重新开始 |
| `blocked` | 阻塞 | 因外部原因暂停 |
| `cancelled` | 已取消 | 可重新激活为 pending |

### 3.2 VALID_TRANSITIONS — 状态流转表

```
draft       → pending, cancelled
pending     → claimed, in_progress, cancelled
claimed     → in_progress, pending, cancelled
in_progress → submitted, blocked, cancelled
submitted   → done, rejected
rejected    → in_progress
blocked     → in_progress, cancelled
done        → （终态，无后继）
cancelled   → pending（可重新激活）
```

**状态流转图**：

```
                    ┌──────────────────────────────────────────────────┐
                    │                  cancelled ◄────────────────────┐│
                    │                     │                           ││
                    │                     ▼ (重新激活)                ││
  ┌───────┐  审阅通过  ┌─────────┐  领取  ┌─────────┐  开始  ┌────────────┐  提交  ┌───────────┐
  │ draft │──────────►│ pending │──────►│ claimed │──────►│in_progress │──────►│ submitted │
  └───────┘          └─────────┘       └─────────┘       └────────────┘       └───────────┘
                          ▲               │  释放             │ ▲                  │
                          │               └──────────────────►│ │                  │
                          │                                   │ │ 重新开始          │
                          │               ┌─────────┐        │ │             ┌────┴────┐
                          │               │ blocked │◄───────┘ │             │         │
                          │               └────┬────┘          │          通过│      驳回│
                          │                    └───────────────┘             ▼         ▼
                          │                                            ┌──────┐  ┌──────────┐
                          │                                            │ done │  │ rejected │
                          │                                            └──────┘  └────┬─────┘
                          │                                                           │
                          └───────────────────────────────────────────────────────────┘
```

### 3.3 ReviewStatus — 审阅状态

| 值 | 含义 |
|----|------|
| `not_required` | 不需要审阅（创建时未指定 reviewer） |
| `pending_review` | 等待审阅 |
| `approved` | 审阅通过 |
| `rejected` | 审阅驳回 |

### 3.4 TestStatus — 测试状态

| 值 | 含义 |
|----|------|
| `not_run` | 未运行 |
| `running` | 运行中 |
| `passed` | 通过 |
| `failed` | 失败 |

### 3.5 TaskType — 任务类型

| 值 | 含义 |
|----|------|
| `master` | 主任务（可拆解为子任务） |
| `subtask` | 子任务（隶属某主任务） |

## 4. 方法详解

### 4.1 任务创建

#### `create_task(title, description="", priority=3, tags=None, task_type=MASTER, parent_id=None, reviewer=None) → Task`

创建一个新任务并持久化到数据库。

**参数**：

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `title` | `str` | — | 任务标题（**必填**，不可为空白） |
| `description` | `str` | `""` | 任务详细描述 |
| `priority` | `int` | `3` | 优先级，1（最高）到 5（最低） |
| `tags` | `list[str] \| None` | `None` | 标签列表，`None` 时默认为 `[]` |
| `task_type` | `TaskType` | `MASTER` | 任务类型 |
| `parent_id` | `str \| None` | `None` | 父任务 ID（创建子任务时必填） |
| `reviewer` | `str \| None` | `None` | 审阅者终端 ID |

**业务规则**：

1. 标题经过 `strip()` 后不能为空，否则抛出 `TaskManagerError`
2. 优先级必须在 `[1, 5]` 范围内
3. 如果指定了 `parent_id`，则该父任务必须存在
4. **审阅分支**：
   - 指定 `reviewer` → 初始 `status=DRAFT`, `review_status=PENDING`
   - 未指定 `reviewer` → 初始 `status=PENDING`, `review_status=NOT_REQUIRED`
5. `created_by` 自动设置为当前 `terminal_id`
6. 操作写入日志，action 为 `"created"`

**返回**：创建成功的 `Task` 对象。

**异常**：

```python
TaskManagerError("任务标题不能为空")
TaskManagerError("优先级必须在 1-5 之间")
TaskManagerError(f"父任务 {parent_id} 不存在")
```

**示例**：

```python
# 创建普通任务（直接进入 pending，可被领取）
task = tm.create_task("实现用户登录", priority=2, tags=["auth", "backend"])

# 创建需要审阅的任务（进入 draft，等待审阅者批准）
task = tm.create_task("重构数据库层", reviewer="master-1")
```

### 4.2 任务拆解

#### `decompose_task(parent_id, subtasks, reviewer=None) → list[Task]`

将一个主任务拆解为多个子任务。

**参数**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `parent_id` | `str` | 父任务 ID |
| `subtasks` | `list[dict]` | 子任务定义列表 |
| `reviewer` | `str \| None` | 统一为所有子任务设置的审阅者 |

**subtasks 中每项 dict 的结构**：

| 键 | 类型 | 必填 | 说明 |
|----|------|------|------|
| `title` | `str` | 是 | 子任务标题 |
| `description` | `str` | 否 | 描述，默认 `""` |
| `priority` | `int` | 否 | 优先级，**默认继承父任务** |
| `tags` | `list[str]` | 否 | 标签，**默认继承父任务**（深拷贝） |

**业务规则**：

1. 父任务必须存在（通过 `_get_or_raise` 校验）
2. 父任务 `task_type` 必须为 `MASTER`，否则抛出 `TaskManagerError("只有主任务可以拆解")`
3. 每个子任务通过内部调用 `create_task()` 创建，`task_type` 强制为 `SUBTASK`
4. 子任务未显式指定 `priority`/`tags` 时，继承父任务对应值
5. 记录父任务日志，action 为 `"decomposed"`

**返回**：创建的子任务列表 `list[Task]`。

**示例**：

```python
subtasks = [
    {"title": "设计 API 接口", "description": "RESTful API 设计"},
    {"title": "实现前端页面", "priority": 2},
    {"title": "编写单元测试", "tags": ["test"]},
]
children = tm.decompose_task(parent.id, subtasks, reviewer="master-1")
# 返回 3 个 Task 对象，均为 subtask 类型
```

### 4.3 任务领取 / 释放

#### `claim_task(task_id) → Task`

当前终端领取一个待执行的任务。

**前置条件**：

- 任务存在
- 任务状态必须为 `pending`

**行为**：

1. 设置 `status = CLAIMED`
2. 设置 `claimed_by = self.terminal_id`
3. 设置 `claimed_at` 为当前时间（ISO 8601）
4. 持久化并记录日志（action: `"claimed"`）

**异常**：

```python
TaskManagerError(f"只能领取 pending 状态的任务，当前状态: {task.status.value}")
```

---

#### `unclaim_task(task_id) → Task`

释放已领取的任务，使其回到待领取池。

**前置条件**：

- 任务状态必须为 `claimed`
- `claimed_by` 必须等于当前 `terminal_id`（**只能释放自己的任务**）

**行为**：

1. 设置 `status = PENDING`
2. 清空 `claimed_by = None`, `claimed_at = None`
3. 持久化并记录日志（action: `"unclaimed"`）

**异常**：

```python
TaskManagerError("只能释放 claimed 状态的任务")
TaskManagerError("只能释放自己领取的任务")
```

---

### 4.4 状态变更

#### `change_status(task_id, new_status) → Task`

通用状态变更方法，严格遵循 `VALID_TRANSITIONS` 状态机。

**参数**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `task_id` | `str` | 任务 ID |
| `new_status` | `TaskStatus` | 目标状态 |

**行为**：

1. 调用 `task.can_transition_to(new_status)` 校验合法性
2. 校验失败时，错误消息中会列出当前状态允许的所有目标状态
3. 更新并持久化
4. 记录日志（action: `"status_changed"`，含 `old_value` 和 `new_value`）

**异常示例**：

```
不能从 done 转为 pending。允许的状态: 无（终态）
不能从 pending 转为 submitted。允许的状态: claimed, in_progress, cancelled
```

> **注意**：`start_task` 和 `submit_task` 是对 `change_status` 的高层封装，附加了更严格的前置校验和额外的字段更新。

---

#### `start_task(task_id) → Task`

开始执行任务。

**允许的前置状态**：`claimed` 或 `rejected`

**状态变更**：`→ in_progress`

**实现**：先校验状态，再委托 `change_status()` 完成实际变更。

**异常**：

```python
TaskManagerError(f"只能从 claimed/rejected 状态开始工作，当前: {task.status.value}")
```

---

#### `submit_task(task_id) → Task`

提交任务等待验收。

**前置状态**：仅 `in_progress`

**状态变更**：`→ submitted`

**额外行为**：设置 `submitted_at` 为当前时间。

> **注意**：`submit_task` 不走 `change_status()` 通道，而是直接修改字段并调用 `db.update_task()`，因为它需要同时设置 `submitted_at` 时间戳。日志 action 为 `"submitted"`。

**异常**：

```python
TaskManagerError("只能提交 in_progress 状态的任务")
```

---

### 4.5 审核验收

#### `verify_task(task_id, approved, comment="") → Task`

主终端对已提交任务进行验收。

**参数**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `task_id` | `str` | 任务 ID |
| `approved` | `bool` | `True` 通过，`False` 驳回 |
| `comment` | `str` | 验收意见 |

**前置状态**：仅 `submitted`

**行为**：

| `approved` | 结果状态 | 日志 action |
|------------|---------|-------------|
| `True` | `done` | `"verified"` |
| `False` | `rejected` | `"rejected"` |

**额外字段更新**（无论通过与否）：

- `verified_by = self.terminal_id`
- `verified_at` = 当前时间
- `verify_comment = comment`

**异常**：

```python
TaskManagerError("只能验收 submitted 状态的任务")
```

---

### 4.6 审阅流程

审阅流程与验收流程**对称设计**，但作用于任务生命周期的不同阶段：

| 维度 | 审阅（Review） | 验收（Verify） |
|------|---------------|---------------|
| 作用阶段 | 创建后、执行前 | 提交后、完成前 |
| 前置状态 | `draft` | `submitted` |
| 通过后状态 | `pending` | `done` |
| 驳回后状态 | `draft`（不变） | `rejected` |
| 可重新提交 | ✅ `resubmit_for_review` | ✅ `start_task` → 重新执行 |

#### `review_task(task_id, approved, comment="") → Task`

审阅任务定义。

**参数**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `task_id` | `str` | 任务 ID |
| `approved` | `bool` | `True` 通过，`False` 驳回 |
| `comment` | `str` | 审阅意见 |

**前置条件**：

- `status == DRAFT`
- `review_status == PENDING`（即 `pending_review`）

**行为**：

| `approved` | 结果 | 日志 action |
|------------|------|-------------|
| `True` | `status=PENDING`, `review_status=APPROVED` | `"review_approved"` |
| `False` | `status` 保持 `DRAFT`, `review_status=REJECTED` | `"review_rejected"` |

两种情况下 `review_comment` 都会设置为传入的 `comment`。

**异常**：

```python
TaskManagerError("只能审阅 draft 状态的任务")
TaskManagerError("该任务未要求审阅或已审阅完成")
```

---

#### `resubmit_for_review(task_id, reviewer=None) → Task`

审阅被驳回后，修改任务定义再重新提交审阅。

**参数**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `task_id` | `str` | 任务 ID |
| `reviewer` | `str \| None` | 新审阅者（留空则沿用原审阅者） |

**前置条件**：

- `status == DRAFT`
- `review_status == REJECTED`

**行为**：

1. 如果传入新的 `reviewer`，更新 `task.reviewer`
2. 重置 `review_status = PENDING`
3. 清空 `review_comment = ""`
4. 持久化并记录日志（action: `"review_resubmitted"`）

**异常**：

```python
TaskManagerError("只能重新提交 draft 状态的任务")
TaskManagerError("只能重新提交被驳回的任务")
```

### 4.7 测试相关

#### `update_test_result(task_id, test_status, test_report=None) → Task`

更新任务的测试执行结果。

**参数**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `task_id` | `str` | 任务 ID |
| `test_status` | `TestStatus` | 测试状态枚举 |
| `test_report` | `dict \| None` | 测试报告（可选），JSON 可序列化的字典 |

**行为**：

1. 更新 `task.test_status`
2. 如果 `test_report is not None`，同时更新 `task.test_report`
3. 记录日志（action: `"test_run"`，含 `old_value` 和 `new_value`）

> **设计说明**：`test_report` 为 `None` 时不更新，这允许仅变更测试状态而保留之前的报告数据。

### 4.8 查询方法

查询方法均为**只读操作**，不记录日志、不修改数据。

#### `get_task(task_id) → Optional[Task]`

根据 ID 获取单个任务。

- 任务存在 → 返回 `Task` 对象
- 任务不存在 → 返回 `None`（**不抛异常**，区别于 `_get_or_raise`）

---

#### `list_tasks(**kwargs) → list[Task]`

列出任务，支持多种过滤条件。直接透传给 `db.list_tasks()`。

**支持的过滤参数**（由 Database 层实现）：

| 参数 | 类型 | 说明 |
|------|------|------|
| `status` | `str` | 按状态过滤 |
| `task_type` | `str` | 按类型过滤（`master` / `subtask`） |
| `parent_id` | `str` | 按父任务过滤 |
| `claimed_by` | `str` | 按领取者过滤 |
| `tag` | `str` | 按标签过滤 |
| `limit` | `int` | 返回数量限制 |

**示例**：

```python
# 列出所有进行中的任务
tasks = tm.list_tasks(status="in_progress")

# 列出某终端领取的所有任务
tasks = tm.list_tasks(claimed_by="worker-1")

# 列出带特定标签的任务，限制 10 条
tasks = tm.list_tasks(tag="backend", limit=10)
```

---

#### `list_subtasks(parent_id) → list[Task]`

列出指定主任务的所有子任务。

**实现**：等价于 `list_tasks(parent_id=parent_id)`。

---

#### `get_progress(parent_id) → dict`

获取主任务的子任务完成进度统计。

**返回结构**：

```python
{
    "parent_id": "abc12345",
    "total": 5,          # 子任务总数
    "done": 3,           # 已完成数
    "progress": 60.0,    # 完成百分比（保留 1 位小数）
    "by_status": {       # 按状态分组计数
        "pending": 1,
        "in_progress": 1,
        "done": 3,
    }
}
```

> 当 `total == 0` 时，`progress` 返回 `0`（避免除零错误）。

---

#### `get_logs(task_id=None, **kwargs) → list[TaskLog]`

获取操作日志。

- 传入 `task_id` → 获取该任务的日志
- 不传 `task_id` → 获取全局日志
- 额外 `kwargs` 透传给 `db.list_logs()` 用于更多过滤

### 4.9 更新 / 删除

#### `update_task(task_id, title=None, description=None, priority=None, tags=None, test_path=None, work_dir=None) → Task`

更新任务的可变属性。

**参数**（全部可选，`None` 表示不更新该字段）：

| 参数 | 类型 | 说明 |
|------|------|------|
| `title` | `str` | 标题（自动 `strip()`） |
| `description` | `str` | 描述（自动 `strip()`） |
| `priority` | `int` | 优先级 |
| `tags` | `list[str]` | 标签列表 |
| `test_path` | `str` | 测试文件路径 |
| `work_dir` | `str` | 工作目录 |

**行为**：

1. 只更新传入的非 `None` 字段
2. 内部记录变更内容列表（如 `["title=xxx", "priority=2"]`）
3. 有变更时记录日志（action: `"updated"`，detail 为分号分隔的变更说明）
4. 无变更时仍调用 `db.update_task()`，但不记录日志

**异常**：任务不存在时抛出 `TaskManagerError`。

---

#### `delete_task(task_id) → bool`

删除任务。

**返回**：

- `True` — 删除成功
- `False` — 任务不存在

**行为**：删除前记录日志（action: `"deleted"`），然后调用 `db.delete_task()`。

> **注意**：此方法在任务不存在时不抛异常，而是静默返回 `False`，与 `get_task` 的行为一致。

### 4.10 终端管理

#### `register_terminal(terminal) → Terminal`

注册或更新一个终端。

**参数**：`Terminal` 数据对象。

**行为**：委托 `db.upsert_terminal()`，已存在则更新，不存在则插入。

---

#### `list_terminals() → list[Terminal]`

列出所有已注册的终端。

### 4.11 内部辅助方法

#### `_get_or_raise(task_id) → Task`

获取任务，不存在则直接抛出异常。

```python
TaskManagerError(f"任务 {task_id} 不存在")
```

> 绝大多数写操作方法的第一步都调用此方法，确保后续逻辑的 `task` 对象一定有效。与 `get_task()` 的区别在于：`get_task` 返回 `Optional`，而 `_get_or_raise` 返回非空值或直接中断。

---

#### `_log(task_id, action, old_value="", new_value="", detail="")`

记录一条操作日志到数据库。

**参数**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `task_id` | `str` | 关联的任务 ID |
| `action` | `str` | 操作类型标识 |
| `old_value` | `str` | 变更前的值 |
| `new_value` | `str` | 变更后的值 |
| `detail` | `str` | 详细说明 |

**自动填充**：

- `terminal_id` = `self.terminal_id`
- `timestamp` = 当前时间（由 `TaskLog` 的 `default_factory` 提供）

**所有 action 值汇总**：

| action | 触发方法 | 说明 |
|--------|---------|------|
| `created` | `create_task` | 任务创建 |
| `decomposed` | `decompose_task` | 主任务拆解 |
| `claimed` | `claim_task` | 任务领取 |
| `unclaimed` | `unclaim_task` | 任务释放 |
| `status_changed` | `change_status` | 通用状态变更 |
| `submitted` | `submit_task` | 任务提交 |
| `verified` | `verify_task` (通过) | 验收通过 |
| `rejected` | `verify_task` (驳回) | 验收驳回 |
| `review_approved` | `review_task` (通过) | 审阅通过 |
| `review_rejected` | `review_task` (驳回) | 审阅驳回 |
| `review_resubmitted` | `resubmit_for_review` | 重新提交审阅 |
| `test_run` | `update_test_result` | 测试执行 |
| `updated` | `update_task` | 属性更新 |
| `deleted` | `delete_task` | 任务删除 |

## 7. 设计要点总结

| 设计决策 | 理由 |
|---------|------|
| 所有写操作记录 TaskLog | 提供完整审计追踪能力 |
| 状态变更严格校验 VALID_TRANSITIONS | 防止非法状态跳转，保障数据一致性 |
| terminal_id 贯穿所有操作 | 支持多终端协作场景下的操作归属追溯 |
| 审阅流程与验收流程对称设计 | 统一的审批模式，降低认知负担 |
| 纯业务逻辑层，不依赖 IO 框架 | 4 个访问层可复用同一套逻辑，易于测试 |
| `_get_or_raise` 内部方法 | 减少重复的空值校验代码 |
| `create_task` 的审阅分支 | 一个方法支持两种工作流（有/无审阅），简化上层调用 |
| 子任务继承父任务的优先级和标签 | 减少重复配置，同时允许覆盖 |
| `delete_task` 不抛异常 | 幂等设计，重复删除不会报错 |

### 9.2 异常处理：防御性编程模式

```python
from cli_anything.core.task_manager import TaskManager, TaskManagerError

tm = TaskManager(db, terminal_id="worker-1")

# ── 创建时校验 ──
try:
    tm.create_task("")  # 空标题
except TaskManagerError as e:
    print(e)  # "任务标题不能为空"

try:
    tm.create_task("任务", priority=0)  # 优先级越界
except TaskManagerError as e:
    print(e)  # "优先级必须在 1-5 之间"

try:
    tm.create_task("任务", parent_id="不存在的ID")
except TaskManagerError as e:
    print(e)  # "父任务 不存在的ID 不存在"

# ── 状态校验 ──
task = tm.create_task("测试任务")  # pending

try:
    tm.start_task(task.id)  # pending 不能直接 start
except TaskManagerError as e:
    print(e)  # "只能从 claimed/rejected 状态开始工作，当前: pending"

try:
    tm.submit_task(task.id)  # pending 不能 submit
except TaskManagerError as e:
    print(e)  # "只能提交 in_progress 状态的任务"

# ── 权限校验 ──
other = TaskManager(db, terminal_id="worker-2")
tm.claim_task(task.id)  # worker-1 领取

try:
    other.unclaim_task(task.id)  # worker-2 无法释放 worker-1 的任务
except TaskManagerError as e:
    print(e)  # "只能释放自己领取的任务"
```

## 10. 边界场景示例

### 10.1 状态机非法跳转——全面测试

以下示例展示了所有会被状态机拒绝的典型非法跳转：

```python
from cli_anything.core.task_manager import TaskManager, TaskManagerError
from cli_anything.core.models import TaskStatus

tm = TaskManager(db, terminal_id="worker-1")

# ── draft 状态的非法操作 ──
task = tm.create_task("草稿任务", reviewer="master-1")  # draft
assert task.status == TaskStatus.DRAFT

for illegal in [TaskStatus.CLAIMED, TaskStatus.IN_PROGRESS,
                TaskStatus.SUBMITTED, TaskStatus.DONE,
                TaskStatus.REJECTED, TaskStatus.BLOCKED]:
    try:
        tm.change_status(task.id, illegal)
        assert False, f"不应该成功: draft → {illegal.value}"
    except TaskManagerError as e:
        print(f"✗ draft → {illegal.value}: {e}")
# ✗ draft → claimed: 不能从 draft 转为 claimed。允许的状态: pending, cancelled
# ✗ draft → in_progress: ...
# ...（draft 只能 → pending 或 cancelled）

# ── done 是终态，不能再转换 ──
task2 = tm.create_task("已完成任务")
tm.claim_task(task2.id)
tm.start_task(task2.id)
tm.submit_task(task2.id)
tm.verify_task(task2.id, approved=True)
assert task2.status == TaskStatus.DONE  # 注意：需重新 get

for target in TaskStatus:
    if target == TaskStatus.DONE:
        continue
    try:
        tm.change_status(task2.id, target)
        assert False, f"done 不应该能转为 {target.value}"
    except TaskManagerError as e:
        print(f"✗ done → {target.value}: {e}")
# 全部失败，错误消息: "允许的状态: 无（终态）"
```

## 11. 任务依赖图（DAG）

### 11.1 概述

P2 引入**任务依赖图**功能，使子任务之间可以声明明确的前置关系（DAG — Directed Acyclic Graph）。系统在 Worker 尝试领取任务时自动检查依赖，未完成的前置任务会阻塞领取。

**核心特性**：

- 声明式依赖：用 `add_dependency(task_id, depends_on_id)` 即可声明依赖关系
- 自动阻塞：`claim_task` 检查所有前置依赖是否为 `done`，未完成则报错
- 循环检测：系统使用 BFS 检测并拒绝形成循环依赖的操作
- 幂等设计：重复添加同一依赖不报错；移除不存在的依赖才报错

### 11.2 三个新方法

#### `add_dependency(task_id, depends_on_id)` → `None`

为任务添加前置依赖关系。`task_id` 任务在 `depends_on_id` 完成（`done`）前无法被领取。

```python
# B 必须等待 A 完成后才能开始
tm.add_dependency(b.id, a.id)
```

**校验规则**：

| 情况 | 行为 |
|------|------|
| 任务不存在 | 抛 `TaskManagerError` |
| 自依赖（task_id == depends_on_id） | 抛 `TaskManagerError` |
| 形成循环依赖 | 抛 `TaskManagerError` |
| 重复添加 | 幂等，不报错 |

**日志 action**：`dep_added`

#### `remove_dependency(task_id, depends_on_id)` → `None`

移除已存在的前置依赖关系。

```python
tm.remove_dependency(b.id, a.id)
```

**异常**：依赖关系不存在时抛 `TaskManagerError`（"任务 X 不存在对 Y 的依赖"）

**日志 action**：`dep_removed`

#### `get_dependencies(task_id)` → `dict`

获取任务的完整依赖关系详情。

```python
info = tm.get_dependencies(task_id)
# {
#   "task_id": "sub-b",
#   "depends_on": [{"id": "sub-a", "title": "子任务A", "status": "pending"}],
#   "depended_by": [{"id": "sub-c", "title": "子任务C", "status": "pending"}],
#   "blocking": ["sub-a"],   # 尚未 done 的前置任务
#   "is_blocked": True
# }
```

| 字段 | 说明 |
|------|------|
| `depends_on` | 该任务依赖的前置任务列表（含状态） |
| `depended_by` | 依赖该任务的下游任务列表 |
| `blocking` | 当前阻塞该任务的前置任务 ID 列表（未 done） |
| `is_blocked` | 是否被阻塞 |

### 11.3 `claim_task` 的依赖检查

```python
def claim_task(task_id):
    # ...
    blocking = self.db.list_blocking_deps(task_id)
    if blocking:
        raise TaskManagerError(
            f"任务 {task_id} 有未完成的前置依赖，无法领取。"
            f"请先完成: {blocking}"
        )
```

**设计意图**：依赖检查只在 `claim` 时触发，不影响 `pending` 状态本身。任务保持 `pending` 可见，但 Worker 领取时会收到明确的阻塞信息。

### 11.4 循环检测算法

使用 BFS（广度优先搜索）从 `depends_on_id` 出发遍历其所有依赖链，若能到达 `task_id` 则说明会形成循环：

```python
# 示例：A→B，B→C，现在尝试 C→A（会形成循环）
tm.add_dependency(b.id, a.id)  # A←B
tm.add_dependency(c.id, b.id)  # A←B←C
tm.add_dependency(a.id, c.id)  # 尝试 C←A，会从 C 遍历到 A → 检测到循环
# raises: TaskManagerError("添加依赖会形成循环：...")
```

### 11.5 数据存储

新建 `task_deps` 表，通过 `_migrate_task_deps_on()` 对已有数据库幂等迁移：

```sql
CREATE TABLE IF NOT EXISTS task_deps (
    task_id     TEXT NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
    depends_on  TEXT NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
    PRIMARY KEY (task_id, depends_on)
);
```

### 11.6 完整使用示例

```python
master = TaskManager(db, terminal_id="master-1")
worker = TaskManager(db, terminal_id="worker-1")

# 创建主任务和 3 个子任务
parent = master.create_task("电商模块", task_type=TaskType.MASTER)
subs = master.decompose_task(parent.id, [
    {"title": "数据库建模"},    # a
    {"title": "接口实现"},     # b（依赖 a）
    {"title": "前端集成"},     # c（依赖 b）
])
a, b, c = subs

# 声明依赖链：a → b → c
master.add_dependency(b.id, a.id)
master.add_dependency(c.id, b.id)

# Worker 尝试直接领取 b（被阻塞）
try:
    worker.claim_task(b.id)
except TaskManagerError as e:
    print(e)  # "任务 xxx 有未完成的前置依赖，无法领取。请先完成: ['a-id']"

# 完成 a
worker.claim_task(a.id)
worker.start_task(a.id)
worker.submit_task(a.id)
master.verify_task(a.id, approved=True)

# 现在 b 可以被领取了
worker.claim_task(b.id)  # ✓

# 查看 c 的依赖状态
info = master.get_dependencies(c.id)
print(info["is_blocked"])   # True（b 还没完成）
print(info["blocking"])      # [b.id]
```

## 12. Judgment Day — 双盲对抗审查

### 11.1 概述

Judgment Day 是 CLI-Anything 的**双盲对抗审查**机制，灵感来自 agent-teams-lite 项目。当 Worker 提交（`submitted`）任务后，Master 可以触发 Judgment Day，系统自动创建两个独立的 `REVIEW` 类型审查任务，分别交由不同 Worker 认领。两位 Judge **互不知晓对方结论**，独立完成审查并提交裁决。最终由 `synthesize_judgment` 综合两份裁决，给出 `approve`/`fix`/`escalated` 三种建议。

**核心优势**：
- 双盲设计消除单一审查者的主观偏差
- confirmed（两者都发现）的问题置信度最高
- suspect（只有一方发现）的问题需要二次判断
- 最多 2 轮自动修复，第 3 轮触发升级（escalated）提醒人工介入

### 11.2 四个新方法

#### `trigger_judgment_day(task_id)` → `tuple[Task, Task]`

为 `submitted` 状态的任务创建双盲审查对。

```python
judge_a, judge_b = tm.trigger_judgment_day(task_id)
# judge_a.tags = ["jd-judge-a", "jd-round-1"]
# judge_b.tags = ["jd-judge-b", "jd-round-1"]
# 两者 task_type = TaskType.REVIEW，status = TaskStatus.PENDING
```

**错误情况**：
- 任务状态不是 `submitted` → `TaskManagerError`
- 已有进行中的审查 → `TaskManagerError`
- 已触发 2 轮仍未通过 → `TaskManagerError`（建议人工介入）

#### `submit_verdict(review_task_id, verdict, findings, summary)` → `Task`

Worker 在审查完成后提交裁决，将结果写入 `test_report`，状态变为 `submitted`。

```python
result = tm.submit_verdict(
    review_task_id=judge_a.id,
    verdict="issues",               # "clean" 或 "issues"
    findings=[
        {
            "desc": "空指针未处理",
            "severity": "CRITICAL",     # CRITICAL / WARNING / SUGGESTION
            "location": "main.py:42",
        }
    ],
    summary="发现 1 个严重问题",
)
# result.test_report = {"verdict": "issues", "findings": [...], "summary": "..."}
# result.status = TaskStatus.SUBMITTED
```

**说明**：
- 接受 `CLAIMED` 或 `IN_PROGRESS` 状态的审查任务（`CLAIMED` 时自动过渡）
- `findings` 为 `None` 或 `[]` 时代表未发现问题

#### `get_review_tasks(task_id)` → `list[Task]`

获取某任务的所有审查任务，按创建时间升序。

```python
reviews = tm.get_review_tasks(task_id)
# 返回该任务下所有 TaskType.REVIEW 的任务（含历史轮次）
```

#### `synthesize_judgment(task_id)` → `dict`

综合最新一轮的双盲审查结果。

```python
result = tm.synthesize_judgment(task_id)
# {
#   "round": 1,
#   "judge_a": {"task_id": "...", "verdict": "issues", "summary": "...", "findings_count": 2},
#   "judge_b": {"task_id": "...", "verdict": "issues", "summary": "...", "findings_count": 2},
#   "confirmed": [{"desc": "空指针未处理", "severity": "CRITICAL", "location": "..."}],
#   "suspect_a": [],
#   "suspect_b": [{"desc": "命名不规范", "severity": "SUGGESTION"}],
#   "both_clean": false,
#   "recommendation": "fix",    # "approve" / "fix" / "escalated"
# }
```

**recommendation 判断逻辑**：
| 条件 | recommendation |
|------|---------------|
| 双方均为 `clean` | `approve`（可以验收） |
| 有问题，且是第 1 轮 | `fix`（修复后进入第 2 轮） |
| 有问题，且是第 2 轮 | `escalated`（需人工判断） |

**findings 匹配算法**：取 `desc` 前 40 字符小写规范化后做精确匹配（简单近似）。

### 11.3 完整使用流程示例

```python
master = TaskManager(db, terminal_id="master-1")
worker = TaskManager(db, terminal_id="worker-1")
judge1 = TaskManager(db, terminal_id="judge-agent-1")
judge2 = TaskManager(db, terminal_id="judge-agent-2")

# 1. Worker 完成并提交任务
task = worker.create_task("实现 OAuth 登录模块")
worker.claim_task(task.id)
worker.start_task(task.id)
worker.submit_task(task.id)

# 2. Master 触发双盲审查
judge_a, judge_b = master.trigger_judgment_day(task.id)
print(f"Judge A: {judge_a.id}, Judge B: {judge_b.id}")

# 3. 两个审查者独立领取、独立审查
judge1.claim_task(judge_a.id)
judge2.claim_task(judge_b.id)

judge1.submit_verdict(
    judge_a.id, "issues",
    findings=[{"desc": "缺少 CSRF 防护", "severity": "CRITICAL", "location": "auth.py:88"}],
    summary="发现 1 个安全漏洞",
)
judge2.submit_verdict(
    judge_b.id, "issues",
    findings=[
        {"desc": "缺少 CSRF 防护", "severity": "CRITICAL", "location": "auth.py:88"},
        {"desc": "日志记录过于详细泄露 token", "severity": "WARNING"},
    ],
    summary="发现 2 个问题",
)

# 4. Master 综合裁决
synthesis = master.synthesize_judgment(task.id)
print(synthesis["recommendation"])   # "fix"
print(synthesis["confirmed"])         # [{"desc": "缺少 CSRF 防护", ...}]
print(synthesis["suspect_b"])         # [{"desc": "日志记录..."}]
```

## 第 13 章 Supervisor 自动路由

### 13.1 背景

当任务池较大、Worker 技能各异时，手动匹配任务与终端效率低下。Supervisor 自动路由通过
任务 tags 与终端 capabilities 的集合交集，实现双向智能推荐。

### 13.2 核心方法

- `update_capabilities(terminal_id, capabilities)` — 更新终端技能列表
- `route_task(task_id)` — Supervisor 视角：为任务找匹配终端
- `suggest_tasks(terminal_id, limit=10)` — Worker 视角：为终端推荐待领取任务

### 13.3 路由算法

**route_task（任务到终端）**
- 任务有 tags -> 返回 capabilities 与 tags 有交集的 Worker 终端
- 任务无 tags -> 返回所有活跃 Worker 终端
- Master 终端永远不出现在候选列表

**suggest_tasks（终端到任务）**
- capabilities 为空 -> 返回全量 pending 任务（按 priority 升序）
- capabilities 非空 -> 先过滤有交集 tag 的任务；无匹配则 fallback 全量
- 只返回 pending 任务，最多 limit 条

### 13.4 数据模型变更

Terminal 新增 capabilities 字段（list[str]），在数据库中存为 JSON 字符串
（TEXT NOT NULL DEFAULT '[]'）。旧库通过 _migrate_task_deps_on 中的 capabilities
分支自动完成幂等迁移。
