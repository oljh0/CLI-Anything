## 1. 模块概述

`utils` 包为 CLI-Anything 提供三个**彼此独立**的工具模块：

| 模块 | 职责 | 行数 |
|---|---|---|
| `config.py` | 配置文件加载、合并、持久化 | ~169 |
| `export_import.py` | 任务数据 JSON 导入 / 导出 | ~121 |
| `terminal.py` | 终端类型检测与唯一 ID 生成 | ~46 |

三个模块不存在互相导入关系，各自服务于项目的不同上层模块。它们共同构成架构图中的 **辅助层（Utils Layer）**，为核心层、访问层和存储层提供通用基础设施。

## 2. config.py — 配置管理

### 2.1 配置文件路径

| 路径 | 说明 |
|---|---|
| `~/.cli-anything/config.yaml` | 用户配置文件默认位置（运行时读写） |
| `config/config.example.yaml` | 项目内的示例配置模板（首次初始化时复制） |

- 模块级常量 `CONFIG_DIR` = `~/.cli-anything/`
- 模块级常量 `CONFIG_FILE` = `~/.cli-anything/config.yaml`
- 示例配置候选路径存储在 `_EXAMPLE_CONFIG_CANDIDATES` 列表中，取相对于源文件向上三级的 `config/config.example.yaml`。

---

### 2.2 常量与默认配置

`_DEFAULT_CONFIG` 是一个模块级字典，包含所有配置段的完整默认值。当用户配置文件不存在或缺少某些字段时，系统会自动回退到这些默认值。默认配置涵盖 8 个顶级段：`database`、`terminal`、`mcp_server`、`dashboard`、`testing`、`display`、`notification`、`aliases`（详见 [2.5 完整默认配置结构](#25-完整默认配置结构)）。

### 2.3 深度合并函数 `_deep_merge`

```python
def _deep_merge(base: dict, override: dict) -> dict
```

模块内部私有函数，用于递归合并两个字典：

- **规则**：`override` 中的值覆盖 `base` 中的对应值。
- **嵌套字典**：若 `base[key]` 和 `override[key]` 都是 `dict`，则递归合并（不会用整个子字典替换，而是逐键覆盖）。
- **非字典值**：直接用 `override[key]` 替换 `base[key]`。
- **新键**：`override` 中存在但 `base` 中不存在的键会被追加。

示例：

```python
base     = {"database": {"path": "default.db", "wal_mode": True}}
override = {"database": {"path": "custom.db"}}
# 结果 = {"database": {"path": "custom.db", "wal_mode": True}}
```

### 2.4 Config 类

```python
class Config:
    def __init__(self, config_path: Optional[str] = None)
```

`Config` 是项目的**核心配置管理器**，被 CLI、MCP Server、Dashboard、TUI、通知器、终端管理器等多个模块引用。

**构造参数**：

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `config_path` | `Optional[str]` | `None` → `CONFIG_FILE` | 配置文件路径；未指定时使用 `~/.cli-anything/config.yaml` |

**内部状态**：

- `self._data: dict` — 合并后的完整配置字典
- `self._loaded: bool` — 是否已加载（用于懒加载）

#### `load() → dict`

加载并合并配置：

1. 以 `_DEFAULT_CONFIG` 的副本作为基础。
2. 如果 `config_path` 指向的文件存在，使用 `yaml.safe_load` 读取用户配置。
3. 调用 `_deep_merge` 将用户配置覆盖到默认配置上。
4. 展开 `database.path` 中的 `~` 为完整绝对路径。
5. 标记 `_loaded = True`，返回合并后的字典。

```python
config = Config()
data = config.load()
# data["database"]["path"] → "C:/Users/xxx/.cli-anything/tasks.db"
```

#### `data` 属性（懒加载）

```python
@property
def data(self) -> dict
```

首次访问时自动调用 `load()`，后续直接返回缓存的 `_data`。所有 `get` / `set` 操作均通过此属性触发懒加载。

#### `get(key, default=None) → Any`

按**点号分隔路径**获取嵌套配置值：

```python
config.get("database.path")          # → "~/.cli-anything/tasks.db"
config.get("mcp_server.sse_port")    # → 8000
config.get("nonexistent.key", 42)    # → 42
```

实现方式：将 `key` 按 `.` 拆分为列表，逐级向下索引字典；任何一级不存在则返回 `default`。

#### `set(key, value) → None`

按点号路径设置配置值，并**自动保存**到文件：

```python
config.set("terminal.role", "master")
# 1. 在内存中设置 _data["terminal"]["role"] = "master"
# 2. 自动调用 save() 持久化到 YAML 文件
```

如果中间路径不存在或不是字典，会自动创建空字典。

#### `save() → None`

将内存中的 `_data` 序列化为 YAML 并写入 `config_path`：

- 自动创建目录（`os.makedirs`）。
- 使用 `yaml.dump` 序列化，配置：`default_flow_style=False`（块格式）、`allow_unicode=True`（支持中文）、`sort_keys=False`（保持插入顺序）。

#### `init_config(role="worker", name="") → str`

首次使用时的配置初始化方法：

1. 创建配置目录 `~/.cli-anything/`。
2. 如果配置文件不存在：
   - 优先从 `_EXAMPLE_CONFIG_CANDIDATES` 中查找并复制示例配置。
   - 若示例配置也不存在，则使用 `_DEFAULT_CONFIG`。
3. 调用 `load()` 加载配置。
4. 设置 `terminal.role` 和 `terminal.name`（如果参数非空）。
5. 调用 `save()` 持久化。
6. 返回配置文件路径。

```python
config = Config()
path = config.init_config(role="master", name="主窗口")
# path → "C:/Users/xxx/.cli-anything/config.yaml"
```

### 2.6 设计要点与已知限制

**设计要点**：

1. **YAML 格式**：选择 YAML 而非 JSON/TOML，兼顾可读性与中文注释支持。
2. **深度合并**：用户只需覆盖需要修改的字段，未指定的字段自动继承默认值，降低配置成本。
3. **路径展开**：`database.path` 中的 `~` 会被自动展开为用户主目录绝对路径。
4. **懒加载**：通过 `data` 属性实现延迟加载，避免不必要的文件 I/O。
5. **写入即保存**：`set()` 方法自动触发 `save()`，保证内存与文件一致。
6. **首次初始化**：`init_config` 支持从示例配置复制，降低首次使用门槛。

**已知限制**：

- **`aliases` 段已接入 CLI**：`main()` 会在 Typer 解析前调用 `_resolve_aliases()`，把 `config.yaml` 中的别名展开为真实命令。
- **无配置校验**：加载时不验证配置值的类型或范围（如端口号是否合法）。
- **单例管理**：`Config` 非单例，多处实例化可能导致内存中的配置状态不同步（需调用方注意）。

## 3. export_import.py — 数据导入导出

### 3.1 导出：export_tasks

```python
def export_tasks(
    tm: TaskManager,
    output_path: str,
    parent_id: Optional[str] = None,
    include_logs: bool = True,
) -> dict
```

将任务数据从数据库导出为 JSON 文件。

**参数**：

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `tm` | `TaskManager` | —（必填） | 任务管理器实例，用于查询任务和日志 |
| `output_path` | `str` | —（必填） | 输出 JSON 文件路径 |
| `parent_id` | `Optional[str]` | `None` | 仅导出指定父任务及其子任务；`None` 表示导出全部 |
| `include_logs` | `bool` | `True` | 是否同时导出操作日志 |

**返回值**：

```python
{
    "tasks_count": 10,      # 导出的任务数量
    "logs_count": 25,       # 导出的日志条数
    "path": "backup.json"   # 输出文件路径
}
```

**执行流程**：

```
1. 确定导出范围
   ├─ parent_id 非空 → 获取父任务 + 所有子任务
   └─ parent_id 为空 → list_tasks(limit=9999) 获取全部任务
2. 构造导出数据结构
   ├─ version: "1.0"
   ├─ exported_at: 当前时间 ISO 格式（精确到秒）
   ├─ tasks: [task.to_dict() ...]
   └─ logs: []（待填充）
3. 如果 include_logs=True
   └─ 遍历每个任务 → tm.get_logs(task.id, limit=999) → 追加到 logs
4. 创建输出目录（os.makedirs）
5. json.dump 写入文件（ensure_ascii=False, indent=2）
6. 返回统计摘要
```

### 3.2 导入：import_tasks

```python
def import_tasks(
    tm: TaskManager,
    input_path: str,
    overwrite: bool = False,
) -> dict
```

从 JSON 文件导入任务数据到数据库。

**参数**：

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `tm` | `TaskManager` | —（必填） | 任务管理器实例 |
| `input_path` | `str` | —（必填） | 输入 JSON 文件路径 |
| `overwrite` | `bool` | `False` | 已存在的任务是否覆盖 |

**返回值**：

```python
{
    "imported": 8,          # 成功导入的任务数
    "skipped": 2,           # 因已存在而跳过的任务数
    "logs_imported": 15,    # 导入的日志条数
    "errors": []            # 错误信息列表
}
```

**执行流程**：

```
1. 读取 JSON 文件 → json.load
2. 遍历 data["tasks"]
   ├─ 获取 task_id → tm.get_task(task_id) 检查是否已存在
   ├─ 已存在且 overwrite=False → skipped += 1, continue
   ├─ 已存在且 overwrite=True  → Task.from_row(td) → tm.db.update_task
   └─ 不存在                     → Task.from_row(td) → tm.db.insert_task
   └─ 异常 → 记录到 errors 列表
3. 遍历 data["logs"]
   ├─ TaskLog.from_row(ld) → 重置 log.id = None（由数据库重新生成）
   └─ tm.db.insert_log(log)
   └─ 异常 → 静默忽略
4. 返回统计摘要
```

**导入模式对比**：

| 行为 | `overwrite=False`（默认） | `overwrite=True` |
|---|---|---|
| 已存在的任务 | 跳过，计入 `skipped` | 覆盖更新 |
| 不存在的任务 | 插入 | 插入 |
| 操作日志 | 总是追加导入 | 总是追加导入 |

> **注意**：当前没有 `mode="replace"`（清空后全量导入）的实现。`overwrite` 参数控制的是逐条覆盖行为，而非全量替换。

### 3.3 JSON 文件格式

导出的 JSON 文件结构如下：

```json
{
  "version": "1.0",
  "exported_at": "2024-01-15T10:30:00",
  "tasks": [
    {
      "id": "uuid-string",
      "title": "任务标题",
      "description": "任务描述",
      "status": "pending",
      "priority": "medium",
      "parent_id": null,
      "assignee": "",
      "created_at": "2024-01-15T10:00:00",
      "updated_at": "2024-01-15T10:00:00"
    }
  ],
  "logs": [
    {
      "id": 1,
      "task_id": "uuid-string",
      "action": "created",
      "detail": "任务创建",
      "terminal_id": "abc12345",
      "created_at": "2024-01-15T10:00:00"
    }
  ]
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `version` | `string` | 导出格式版本号，当前为 `"1.0"` |
| `exported_at` | `string` | ISO 8601 格式的导出时间（精确到秒） |
| `tasks` | `array` | 任务对象数组，每个元素由 `Task.to_dict()` 生成 |
| `logs` | `array` | 操作日志数组，每个元素由 `TaskLog.to_dict()` 生成 |

---

### 3.4 设计要点与已知限制

**设计要点**：

1. **仅支持 JSON 格式**：使用标准库 `json` 模块，无额外依赖。
2. **按 ID 去重**：`overwrite=False` 模式下，通过 `tm.get_task(task_id)` 检查已存在的任务并跳过，避免重复导入。
3. **日志 ID 重置**：导入时将 `log.id` 置为 `None`，由数据库自增机制重新分配，避免 ID 冲突。
4. **容错设计**：任务导入逐条 try/except，单条失败不影响其他任务；日志导入失败静默忽略。
5. **支持部分导出**：通过 `parent_id` 参数可仅导出某个父任务及其子任务树。

**已知限制**：

- **仅通过 CLI 暴露**：导入导出功能仅在 `cli.py` 中以延迟导入方式调用，MCP Server 和 Dashboard 均未集成此功能。
- **无版本兼容性检查**：导入时不校验 `version` 字段，不同版本格式可能导致字段缺失。
- **日志导入无关联校验**：日志中的 `task_id` 不校验对应任务是否存在。
- **全量查询上限**：`list_tasks(limit=9999)` 使用硬编码上限而非分页，任务量极大时可能有内存压力。

## 4. terminal.py — 终端检测

### 4.1 detect_terminal_type

```python
def detect_terminal_type() -> str
```

自动检测当前终端环境类型。

**返回值**：

| 返回值 | 触发条件 |
|---|---|
| `"ssh"` | 环境变量 `SSH_CONNECTION` 或 `SSH_CLIENT` 存在 |
| `"wsl"` | `platform.uname().release` 包含 `"microsoft"`（不区分大小写） |
| `"powershell"` | Windows 平台且 `PSModulePath` 环境变量非空 |
| `"cmd"` | Windows 平台且非 PowerShell |
| `"zsh"` | 非 Windows 平台，`SHELL` 环境变量包含 `"zsh"` |
| `"bash"` | 非 Windows 平台，`SHELL` 环境变量包含 `"bash"` |
| `"unknown"` | 以上条件均不满足 |

**检测优先级**（从高到低）：

```
SSH → WSL → (Windows: PowerShell / CMD) → (Unix: zsh / bash) → unknown
```

> **说明**：SSH 检测优先于所有其他类型，即在 SSH 会话中即使使用 PowerShell，也会返回 `"ssh"`。

### 4.2 generate_terminal_id

```python
def generate_terminal_id() -> str
```

生成一个 8 位十六进制的终端唯一标识符。

**生成逻辑**：

1. 拼接原始字符串：`"{hostname}-{pid}-{TERM_SESSION_ID}"`
   - `hostname`：`platform.node()` 获取主机名
   - `pid`：`os.getpid()` 获取当前进程 PID
   - `TERM_SESSION_ID`：环境变量（macOS Terminal.app 等终端会设置此变量），不存在时为空字符串
2. 对原始字符串进行 **MD5** 哈希
3. 取哈希结果的**前 8 位**作为终端 ID

```python
generate_terminal_id()
# → "a3f1b2c4"（示例）
```

**唯一性保证**：

- **PID** 确保同一台机器上不同终端窗口的 ID 不同。
- **hostname** 确保不同机器的 ID 不同。
- **TERM_SESSION_ID** 为可选的额外区分因子。
- 8 位十六进制提供 16^8 ≈ 43 亿种组合，实际使用中碰撞概率极低。

### 4.3 get_current_pid

```python
def get_current_pid() -> int
```

返回当前进程的 PID（`os.getpid()` 的简单封装）。被 `terminal_manager.py` 在注册终端时使用。

### 4.4 设计要点

1. **跨平台兼容**：通过 `sys.platform` 区分 Windows 与 Unix-like 系统，对 WSL 有专门检测。
2. **纯环境变量检测**：不依赖外部命令或进程树遍历，执行速度快。
3. **确定性 ID**：同一终端窗口中多次调用 `generate_terminal_id()` 返回相同结果（因为 PID 不变），适合在终端注册场景中使用。
4. **轻量级**：仅依赖标准库（`os`、`platform`、`sys`、`hashlib`），无第三方依赖。

## 5. 模块间依赖关系

三个工具模块被项目中的以下上层模块引用：

### 5.1 config.py — 被广泛引用

```
cli.py                     ─→ Config  （CLI 入口，加载全局配置）
mcp_server/server.py       ─→ Config  （MCP 服务器启动时读取配置）
web/dashboard.py           ─→ Config  （Dashboard 读取端口、刷新间隔等）
tui/app.py                 ─→ Config  （TUI 界面读取显示配置）
notification/notifier.py   ─→ Config  （通知器读取通知开关）
core/terminal_manager.py   ─→ Config  （终端管理器读取终端角色等）
```

### 5.2 export_import.py — 仅 CLI 引用

```
cli.py  ─→ export_tasks  （延迟导入，在 export 命令中调用）
cli.py  ─→ import_tasks  （延迟导入，在 import 命令中调用）
```

> 导入导出功能采用**延迟导入**（在函数体内 `from ... import`），避免在不使用时加载多余模块。

### 5.3 terminal.py — 被终端管理器引用

```
core/terminal_manager.py  ─→ detect_terminal_type   （注册终端时检测类型）
core/terminal_manager.py  ─→ generate_terminal_id   （注册终端时生成 ID）
core/terminal_manager.py  ─→ get_current_pid        （注册终端时记录 PID）
```

## 6. 已知限制与未来演进

| 编号 | 模块 | 限制描述 | 影响 | 可能的改进方向 |
|---|---|---|---|---|
| 1 | config.py | 配置文件仍是单文件 YAML | 多终端共享同一配置时需通过 `CLI_ANYTHING_CONFIG` 或终端环境变量隔离身份 | 按项目或终端提供 profile 管理 |
| 2 | config.py | 无配置值校验 | 非法端口号等不会报错 | 增加 schema 校验（如 pydantic） |
| 3 | config.py | 非单例设计 | 多实例间可能状态不同步 | 引入全局单例或配置中心 |
| 4 | export_import.py | 仅 CLI 暴露 | MCP / Dashboard 无法触发导入导出 | 在 MCP 工具中添加 export/import |
| 5 | export_import.py | 无版本兼容性检查 | 跨版本导入可能字段缺失 | 导入时校验 version 并做迁移 |
| 6 | export_import.py | 全量查询 limit=9999 | 超大数据量时内存压力 | 改为分页迭代导出 |
| 7 | terminal.py | SSH 优先级最高 | SSH 下无法区分具体 shell 类型 | 增加二级检测（ssh+bash） |
| 8 | terminal.py | MD5 取前 8 位 | 理论碰撞风险极低但非零 | 如需更高唯一性可使用 UUID |
