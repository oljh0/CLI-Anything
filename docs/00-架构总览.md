## 1. 项目简介

CLI-Anything 是一个**跨终端协同任务系统**，采用 **Master/Worker 协同模式**，在本机多个终端窗口之间协同管理开发任务。

核心理念是：

- **主终端 (Master)** 负责创建、拆解、审阅和验收任务；
- **从终端 (Worker)** 负责领取、实现和提交任务；
- **AI Agent** 可通过 MCP Server 作为特殊 Worker 参与协同。

系统提供 **4 个访问层**（CLI / MCP Server / Web Dashboard / TUI），它们共享同一个 `TaskManager` 核心，底层使用 SQLite WAL 模式数据库保证并发安全。

## 3. 分层设计

### 3.1 访问层 (Access Layer)

访问层是用户与系统交互的入口，共有 4 种形态，满足不同场景的使用需求：

| 访问层 | 技术栈 | 适用场景 | 说明 |
|--------|--------|----------|------|
| **CLI** | Typer + Rich | 终端命令行操作 | 30+ 条命令，支持 Rich 美化输出，是最主要的交互方式 |
| **MCP Server** | FastMCP | AI Agent 协同 | 31 个 MCP 工具，AI Agent 通过 stdio 或 SSE 传输协议调用 |
| **Web Dashboard** | FastAPI + WebSocket | 可视化看板 | 浏览器中查看任务状态，内嵌 HTML 模板 |
| **TUI** | Textual | 终端内图形界面 | DataTable 看板，无需浏览器即可查看全局状态 |

所有访问层共享同一个 `TaskManager` 实例作为业务逻辑入口，保证数据一致性。每个访问层都采用**懒初始化**模式（`_init()` 延迟初始化全局组件），避免不必要的资源开销。

### 3.2 核心层 (Core Layer)

核心层封装了全部业务逻辑，是系统的心脏：

| 组件 | 职责 |
|------|------|
| **TaskManager** | 统一业务入口：任务 CRUD、状态变更、审阅、测试调度 |
| **TerminalManager** | 终端注册、角色分配（Master/Worker）、心跳管理 |
| **HealthChecker** | 终端存活检测、超时任务自动释放回待认领状态 |
| **TestRunner** | pytest 集成，通过子进程运行测试并解析 JSON 格式报告 |
| **Models** | 数据模型定义（Task / TaskLog / Terminal）、状态枚举、状态机 |

### 3.3 存储层 (Storage Layer)

存储层基于 **SQLite WAL 模式**，提供轻量级但支持并发读的持久化方案：

- **`tasks` 表** — 任务主表，包含标题、描述、状态、优先级、指派终端等字段
- **`task_logs` 表** — 任务操作日志，记录每次状态变更的时间、操作者和备注
- **`terminals` 表** — 已注册终端列表，记录终端 ID、角色、最后心跳时间

`Database` 类封装了所有 SQL 操作（仓库模式），上层模块无需直接接触 SQL 语句。

### 3.4 辅助层 (Utils Layer)

辅助层提供跨模块共享的基础设施：

| 组件 | 职责 |
|------|------|
| **Config** | YAML 配置加载、深合并、支持点号路径访问（如 `db.path`） |
| **Notifier** | 系统通知推送，支持 Windows / macOS / Linux 原生通知 |
| **ExportImport** | JSON 格式的任务批量导入导出 |
| **Terminal** | 终端类型自动检测、唯一 ID 生成 |

## 5. 设计模式

CLI-Anything 在架构中运用了以下设计模式：

### 5.1 Master/Worker 模式

系统的核心协同模式。终端分为两种角色：

- **Master（主终端）**：负责任务的创建、拆解、审阅和验收。通常由项目负责人操作。
- **Worker（从终端）**：负责任务的领取、实现和提交。可以是人工操作的终端，也可以是 AI Agent。

这种模式天然适配"人机协同"场景——Master 拆解需求，Worker（人或 AI）分头实现。

### 5.2 有限状态机 (Finite State Machine)

任务生命周期由 **9 种状态**和严格的 `VALID_TRANSITIONS` 表驱动。任何状态变更都必须经过状态机校验，非法流转会被拒绝。这确保了任务流程的可预测性和数据一致性。

### 5.3 仓库模式 (Repository Pattern)

`Database` 类封装了所有 SQL 操作，上层模块（如 `TaskManager`）只与 Python 对象交互，不直接编写 SQL。这使得未来更换存储引擎（如迁移到 PostgreSQL）的成本最小化。

### 5.4 外观模式 (Facade Pattern)

`TaskManager` 是所有访问层的**统一业务入口**。无论是 CLI、MCP Server、Web Dashboard 还是 TUI，都通过 `TaskManager` 的方法来执行业务操作。这避免了各访问层直接耦合底层实现。

### 5.5 懒初始化 (Lazy Initialization)

各访问层都使用 `_init()` 函数延迟初始化全局组件（数据库连接、管理器实例等）。只有在实际需要时才创建资源，避免启动时的不必要开销。

## 6. 状态机

### 6.1 9 种任务状态

| 状态 | 含义 | 说明 |
|------|------|------|
| `draft` | 草稿 | 任务刚创建，尚未审阅 |
| `pending` | 待认领 | 审阅通过，等待 Worker 领取 |
| `claimed` | 已认领 | Worker 已领取，尚未开始工作 |
| `in_progress` | 进行中 | Worker 正在实现任务 |
| `submitted` | 已提交 | Worker 完成实现，等待 Master 验收 |
| `done` | 已完成 | Master 验收通过（**终态，不可变更**） |
| `rejected` | 已驳回 | Master 验收不通过，需要重新实现 |
| `blocked` | 已阻塞 | 任务遇到阻塞，暂时无法继续 |
| `cancelled` | 已取消 | 任务被取消（可重新激活） |

### 6.2 合法状态流转

以下是所有合法的状态流转路径：

```
draft       → pending      （审阅通过）
draft       → cancelled    （放弃草稿）

pending     → claimed      （Worker 领取）
pending     → in_progress  （直接开始，跳过领取）
pending     → cancelled    （取消任务）

claimed     → in_progress  （开始工作）
claimed     → pending      （释放任务，退回待认领）
claimed     → cancelled    （取消任务）

in_progress → submitted    （提交实现）
in_progress → blocked      （遇到阻塞）
in_progress → cancelled    （取消任务）

submitted   → done         （验收通过 ✅ 终态）
submitted   → rejected     （验收驳回）

rejected    → in_progress  （重新开始实现）

blocked     → in_progress  （阻塞解除，继续工作）
blocked     → cancelled    （取消任务）

cancelled   → pending      （重新激活）

done        →（终态，不可变更）
```

## 7. 数据流

### 7.1 典型请求流程

```
用户命令 / AI Agent 调用
        │
        ▼
  ┌─────────────────────────┐
  │  访问层                  │
  │  CLI / MCP / Web / TUI  │
  │  (参数解析、格式化输出)   │
  └───────────┬─────────────┘
              │
              ▼
  ┌─────────────────────────┐
  │  TaskManager             │
  │  (业务逻辑 + 状态校验)    │
  │  ① 参数校验              │
  │  ② 状态机流转检查         │
  │  ③ 调用 Database 持久化   │
  └───────────┬─────────────┘
              │
              ▼
  ┌─────────────────────────┐
  │  Database (SQLite WAL)   │
  │  (SQL 执行 + 事务管理)    │
  └───────────┬─────────────┘
              │
              ▼
  返回 Task / TaskLog / Terminal 对象
```

### 7.2 典型操作示例

以"Worker 领取任务"为例，完整数据流如下：

1. **CLI 层**：用户输入 `task claim <task_id>`，Typer 解析参数
2. **TaskManager**：调用 `claim_task(task_id, terminal_id)` 方法
   - 从数据库获取任务当前状态
   - 检查状态机：`pending → claimed` 是否合法 ✅
   - 更新任务状态和指派终端
3. **Database**：执行 `UPDATE tasks SET status='claimed', assigned_terminal=? WHERE id=?`
4. **Database**：写入 `task_logs` 记录本次操作
5. **CLI 层**：Rich 格式化输出操作结果

## 9. 已知限制

以下是当前版本（0.1.0）已知的设计限制和待改进项：

| # | 限制描述 | 影响范围 | 建议改进方向 |
|---|---------|---------|-------------|
| 1 | WebSocket `broadcast()` 只覆盖 Dashboard API 操作 | Dashboard 无法实时感知 CLI 端的操作 | 在 TaskManager 中加入事件回调，CLI 操作后触发 broadcast |
| 2 | 导入导出功能未暴露为 MCP 工具 | AI Agent 无法执行批量任务导入导出 | 在 MCP Server 中新增 `import_tasks` / `export_tasks` 工具 |
| 3 | Dashboard 前端仍为内嵌 HTML | 前端复杂度继续增长时维护成本较高 | 拆分静态资源或引入轻量构建流程 |
